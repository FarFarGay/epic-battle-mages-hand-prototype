# Спецификация: Прототип Руки — Epic Battle Mages

## 1. Обзор

**Проект:** Прототип механики руки для игры "Epic Battle Mages"
**Технологии:** Vanilla HTML5 Canvas 2D, JavaScript ES Modules (без зависимостей, без сборщика)
**Файл:** `index.html` + модули в `js/` (~7950 строк суммарно)
**Запуск:** Только через локальный HTTP-сервер (ES modules требуют HTTP): `python3 -m http.server 8080`

Прототип реализует основную механику игры — руку волшебника, которая служит курсором. Рука может подбирать, переносить и бросать предметы с реалистичной физикой (гравитация, отскоки, трение, скольжение). Также реализовано управление миньонами через флаг.

```
index.html              — HTML-оболочка (27 строк), загружает js/main.js

├── constants.js        — физические константы, ITEM_TYPES (без начальных позиций)
├── sprites.js          — пиксельные спрайты (*_PIXELS массивы) + спрайты декораций (DECO_*)
├── noise.js            — SimplexNoise 2D, fbm (fractal Brownian motion), createRNG (mulberry32)
├── mapGenerator.js     — generateMap(), placeResources(), placeDecorations(), lastVillages — процедурная генерация
├── VillageGenerator.js — generateVillages() — генерация 4 деревень с зонами производства

├── decorations.js      — decorations[], decoParticles[], onTileChanged(cause), destroyDecorationWithEffect(), addDecorationsToRenderList(), getHarvestType(), harvestDecoration(), setItemSpawnCallback() — дроп ресурсов при разрушении тайлов
├── isometry.js         — изометрическая проекция, camera, screenToIso/isoToScreen, worldToScreen (с высотой)
├── renderer.js         — canvas/ctx, drawPixel, drawPixelArt (с кэшем offscreen canvas), drawIsoDiamond, тени
├── Map.js              — class GameMap: размер, типы тайлов, heightMap, colorCache, seed, туман войны, castlePos
├── GameObject.js       — базовый класс: общая физика (lifting→carried→thrown→…)
├── Item.js             — class Item extends GameObject: логика и отрисовка предмета
├── Minion.js           — class Minion extends GameObject: AI, HP, смерть, отрисовка
├── Hand.js             — class Hand: движение руки, захват, бросок, shake-детектор
├── Castle.js           — class Castle: коллизия, рендер, пушка с отдачей, spawnPoint
├── Fireball.js         — class Fireball extends GameObject: физика огненного шара, взрыв
├── SpellProjectile.js  — class SpellProjectile extends GameObject: бросаемый снаряд для water/earth/wind
├── World.js            — items[], minions[], commandMarkers[], castle, fireball, spellProjectile, screenShake, productionZones[], findZoneAtTile(), resolveCollisions, restartMap, initWorld
├── input.js            — все обработчики событий мыши и клавиатуры
└── main.js             — игровой цикл: update(), render(), gameLoop()
```

### Иерархия классов

```
GameObject (базовая физика)
├── Item            (ресурсы: пшеница, камень, дерево, железо, свиток)
├── Minion          (гоблины: AI блуждания, HP, мёртвое состояние → скелет)
├── Fireball        (огненный шар: взрыв при приземлении, горящие патчи)
└── SpellProjectile (снаряд water/earth/wind: общий объект, spellType на лету)

Hand         (рука-курсор, не наследует GameObject)
Castle       (замок: коллизия, рендер, пушка — не наследует GameObject)
```

**`GameObject`** содержит метод `updatePhysics(dt, hand, triggerShake)` — общую стейт-машину физики для состояний `lifting`, `carried`, `thrown`, `bouncing`, `sliding`. Хуки для переопределения в подклассах:
- `onLand(impactVz)` — при приземлении (используется в `Minion` для урона от падения)
- `onSettle()` — после таймаута settling (переход в `free` или `dead`)

### Рендеринг
- **Изометрическая проекция** — угол 45° (`ISO_ANGLE = π/4`)
- **Тайловая сетка** — `TILE_W = 64px`, `TILE_H ≈ 45px`
- **Pixel-art** — масштаб `PIXEL_SCALE = 3`, сглаживание отключено (`imageSmoothingEnabled = false`). `drawPixelArt` кэширует спрайты на offscreen canvas (Map: pixels→scale→canvas), `ctx.drawImage` вместо попиксельного `fillRect`
- **Глубинная сортировка** — по сумме `ix + iy` для корректного порядка отрисовки
- **Рука** — всегда рендерится последней (`depth: Infinity`), поверх всех объектов

### Game Loop
- `requestAnimationFrame` с вычислением `dt` (delta time)
- Ограничение `dt` до 0.05с (20 FPS минимум) для стабильности физики
- Цикл: `update(dt)` → `render()`

### Стейт-машины
- **Рука:** `open` → `closing` → `closed` → `opening` → `open`
- **Предметы:** `idle` → `lifting` → `carried` → `thrown` → `bouncing` → `sliding` → `settling` → `idle`
- **Миньоны (геймплей):** `free` ↔ `moving_to_point` (ПКМ команда → arrive → `free`); `free` → `busy` (gather команда)
- **Миньоны (физика):** любое → `lifting` → `carried` → `thrown` → `bouncing` → `sliding` → `settling` → `free` (или `dead`)

## 3. Механика руки

### Движение
- Рука следует за курсором мыши с плавной интерполяцией (`lerp = 1 - 0.001^dt`)
- Скрытый системный курсор (`cursor: none`), заменён красной точкой 3×3px

### Спрайты
- **Открытая ладонь** (`HAND_OPEN_PIXELS`) — 10×12 px, вид сверху-сбоку
- **Закрытая ладонь** (`HAND_CLOSED_PIXELS`) — пальцы загнуты внутрь
- Анимация перехода: покадровая интерполяция при `closing`/`opening`
- Покачивание при захвате (sin/cos колебания)

### Захват и бросок
1. **ЛКМ зажать** — если есть предмет в радиусе 1.5 iso-единиц → захват
2. Предмет плавно поднимается к руке (состояние `lifting`)
3. Предмет следует за рукой (состояние `carried`)
4. **ЛКМ отпустить** — бросок с физикой
5. Можно ловить предметы в воздухе (если высота `iz < 2.0`)

### Трекинг скорости
- Хранение последних 10 кадров скорости руки (`velocityHistory`)
- Взвешенное усреднение (последние кадры важнее)
- Учёт массы предмета при расчёте скорости броска

## 4. Физическая система

### Константы

| Константа | Значение | Описание |
|-----------|----------|----------|
| `GRAVITY` | 8.0 | Ускорение свободного падения (ед/с²) |
| `CARRY_HEIGHT` | 1.5 | Высота переноса предмета |
| `LIFT_SPEED` | 4.0 | Скорость подъёма предмета |
| `THROW_VZ_BASE` | 3.0 | Базовый вертикальный импульс при броске |
| `THROW_SCALE` | 1.5 | Множитель скорости руки → скорость броска |
| `BOUNCE_MIN_VZ` | 0.3 | Минимальная вертикальная скорость для отскока |
| `MAX_BOUNCES` | 5 | Максимальное количество отскоков |
| `SLIDE_STOP` | 0.05 | Порог остановки скольжения |
| `HEIGHT_TO_SCREEN` | 40 | Пикселей на единицу высоты |
| `VELOCITY_HISTORY` | 10 | Кадров для усреднения скорости |
| `MAX_THROW_SPEED` | 8.0 | Максимальная скорость броска (iso ед/с) |
| `WALL_BOUNCE` | 0.6 | Коэффициент отскока от стен |
| `CAMERA_OFFSET_Y` | 40 | Вертикальное смещение центра мира (px) |
| `AIR_RESISTANCE` | 0.99 | Коэффициент сопротивления воздуха (за 1/60 сек) |
| `MINION_SPEED` | 1.5 | Скорость блуждания миньона (iso ед/с) |

### Стейт-машина предметов

```
idle → lifting → carried → thrown → bouncing → sliding → settling → idle
                                ↘ sliding → settling → idle
```

| Состояние | Поведение |
|-----------|-----------|
| `idle` | На земле, `iz=0`, можно захватить |
| `lifting` | Плавный подъём к `CARRY_HEIGHT`, easeOut анимация, движение к руке |
| `carried` | Следует за рукой, `iz = CARRY_HEIGHT` |
| `thrown` | Гравитация тянет вниз, горизонтальное движение, сопротивление воздуха (`Math.pow(AIR_RESISTANCE, dt * 60)`) |
| `bouncing` | Отскок: `vz` инвертируется × `bounciness`, горизонтальная скорость × `friction` |
| `sliding` | На земле, экспоненциальное трение замедляет `vx/vy` |
| `settling` | Пауза 0.15с перед возвратом в `idle` |

### Тряска экрана
- При первом ударе предмета о землю
- Интенсивность = `mass × |vz| × 0.5`
- Экспоненциальное затухание
- Максимальная интенсивность: 15px

| 2 | Дерево  | 1.0 | 0.3 | 0.75 | 0.35 | Строительство |
| 3 | Железо  | 3.0 | 0.2 | 0.80 | 0.40 | Снаряжение гоблинов; самый тяжёлый ресурс |
| 4 | Свиток  | 0.3 | 0.1 | 0.85 | 0.25 | Магия; лёгкий, далеко летит |

### Pixel-art спрайты
- Пшеница: 7×8 px — сноп с колосками и перевязкой
- Камень: 7×6 px — изометрический куб (серые тона)
- Дерево: 7×6 px — срез бревна с древесными кольцами и корой
- Железо: 7×6 px — рудный куб (серый с рыжими прожилками)

### Расположение на карте

Ресурсы размещаются процедурно через `placeResources(seed)` в `mapGenerator.js` — биомно-кластерный алгоритм с seeded RNG (воспроизводимо для одного seed).

| Ресурс | typeIndex | Биом | Кол-во | Кластер | Условие |
|--------|-----------|------|--------|---------|---------|
| Пшеница | 0 | plain + village | 30–40 | r=2, до 4 шт. | — |
| Камень | 1 | stone + граница stone↔plain | 25–35 | r=2, до 3 шт. | — |
| Дерево | 2 | forest + граница forest↔plain | 25–35 | r=2, до 4 шт. | — |
| Железо | 3 | stone (глубокий) | 10–15 | r=1, до 2 шт. | dist² > 400 от замка |

## 6. Камера

### Зум
- **Z** (или **Я**) — отдалить на 10% (×0.9), минимум 0.2×
- **X** (или **Ч**) — приблизить на 10% (×1.1), максимум 3.0×
- **C** (или **С**) — сброс зума к 1.0×
- Плавная интерполяция (`dt × 8`)

### Панорамирование (Edge Scroll)
- Камера автоматически движется когда рука подходит к краю экрана
- Зона срабатывания: 12% ширины/высоты экрана с каждого края (`EDGE_ZONE = 0.12`)
- Максимальная скорость: 600 px/с в screen space (`PAN_SPEED = 600`)
- Линейный спад скорости: чем глубже рука в зону — тем быстрее движение
- `camera.x/y` хранят смещение в screen-пикселях
- Все функции `screenToCanvas` во всех модулях учитывают `camera.x/y` — объекты в руке всегда следуют за курсором

### Рендеринг камеры
- Зум через `ctx.scale()` относительно центра экрана
- Панорамирование через `ctx.translate(-camera.x, -camera.y)` перед зумом
- Тряска через `ctx.translate()` поверх зума
- Вертикальное смещение центра мира задаётся константой `CAMERA_OFFSET_Y = 40` (используется в `screenToIso` и `worldToScreen`)
- Индикатор зума в левом нижнем углу (показывается при отклонении от 100%)

## 7. Карта (`Map.js`)

Класс `GameMap` — единственный источник истины о размере поля, тайлах, тумане войны и начальных позициях. Синглтон `gameMap` импортируется в `World.js` и `main.js`.

### Размер и границы

- `gameMap.size = 80` → поле 161×161 тайл (от -80 до +80 по обеим осям)
- Изменение в рантайме: `gameMap.resize(newSize)` — очищает данные вне новых границ
- Невидимые стены по краям поля (`size + 0.5`), предметы отскакивают с `WALL_BOUNCE = 0.6`

### Типы тайлов

Каждый тайл хранит ключ типа (строка). Доступные типы в `TILE_TYPES`:

| Ключ | Цветовая гамма | Анимация | Биом / происхождение |
|------|----------------|----------|----------------------|
| `plain` | Тёмно-фиолетовый (по умолчанию) | — | Генератор |
| `forest` | Тёмно-зелёный | — | Генератор |
| `water` | Тёмно-синий | ✓ волны | Генератор |
| `stone` | Тёмно-серый | — | Генератор |
| `village` | Коричневый | — | Генератор (старые 3×3 кластеры) |
| `village_square` | Светло-коричневый `#c4a46a` | — | VillageGenerator: центр деревни 2×2 |
| `village_house` | Тёмно-коричневый `#7a5c3a` | — | VillageGenerator: тайл с домом |
| `village_road` | Бежевый `#b8a080` | — | VillageGenerator: дорога/мощёная площадь |
| `farmland` | Жёлто-зелёный `#8a9a3a` | — | VillageGenerator: поле с посевами |
| `farmland_ripe` | Золотой `#ccaa22` | ✓ пульсация | VillageGenerator: созревший урожай |
| `mine_tile` | Тёмно-серый `#4a4a55` | — | VillageGenerator: шахта |
| `lumber_tile` | Коричнево-зелёный `#5a6a3a` | — | VillageGenerator: лесоповал |
| `ice` | Холодный синий | — | Генератор |
| `burning` | Оранжевый | ✓ мерцание | Заклинание fire |
| `scorched` | Тёмно-коричневый | — | После горения |
| `puddle` | Синий | ✓ волны | Заклинание water |
| `steam` | Серо-голубой | ✓ | water + fire |
| `swamp` | Болотный зелёный | — | earth + water |
| `wall` | Серо-синий | — | (будущая механика строительства) |
| `rubble` | Светло-серый | — | След валуна / взрыв стены |

API: `gameMap.setTile(ix, iy, 'forest')` / `gameMap.getTile(ix, iy)`

При вызове `setTile()` — автоматически вызывается `_onTileChangedCb(ix, iy, oldType, newType)` (зарегистрирован через `setTileChangedCallback` в `World.js`) и сбрасывается `_colorCache[key]`.

### Туман войны

Состояния тайлов (`FOG`): `hidden` → `explored` → `visible`

| Состояние | Визуал |
|-----------|--------|
| `hidden` | Чёрный тайл (`#0a0a12`) |
| `explored` | Тёмно-серый (посещено, но вне обзора) |
| `visible` | Нормальный цвет типа тайла |

Туман обновляется каждый кадр через `gameMap.tickFog(sources)`:
- Все `visible` → `explored` (через dirty-flag Set `_visibleKeys` — итерация только по видимым, не по всем тайлам)
- Затем `revealAround(ix, iy, radius)` для каждого источника (добавляет в `_visibleKeys`)

**Источники видимости** (радиус в тайлах):
| Источник | Радиус | Форма | Условие |
|----------|--------|-------|---------|
| Замок | 10 (21×21 тайл) | `square` | всегда |
| Живой миньон (basic/warrior) | 2 | `circle` | `state !== 'carried' && state !== 'lifting'` |
| Живой разведчик | 5 | `circle` | `goblinClass === 'scout'`, те же условия |
| Мёртвый миньон | 2 → 0 за 10с | `circle` | `deadTime < 10` — плавное угасание |
| Огненный шар (в полёте) | 3 | `circle` | `state === 'thrown' \|\| state === 'bouncing'` |

**Правила тумана для миньонов:**
- Рука **собственной видимостью не обладает** — поднять предмет вне зоны видимости невозможно.
- Гоблин **в руке** (`carried` / `lifting`) туман **не раскрывает** — водить гоблином по карте как разведчиком нельзя.
- После **смерти** гоблина его зона видимости плавно сужается от 2 до 0 за **10 секунд** (поле `deadTime` растёт в state `dead`).

**Ограничения руки вне зоны видимости:**
- Рука **не может поднимать** предметы или миньонов вне `FOG.VISIBLE` (hover detection пропускает невидимые тайлы).
- Если рука **выходит за пределы** `FOG.VISIBLE` с уже захваченным предметом или миньоном — она **немедленно роняет** его с нулевой скоростью (свободное падение на месте). Это предотвращает разведку картой через перенос гоблина.
- **Исключение — брошенный гоблин:** если гоблин **заброшен в темноту** и выжил при падении, он становится дружественным источником видимости (радиус 2). В его зоне видимости рука **может** подобрать как самого гоблина, так и любые предметы рядом с ним. Это намеренная игровая механика — бросок гоблина как «разведывательный» риск.

Объекты вне `FOG.VISIBLE` скрыты и недоступны для взаимодействия (hover, захват). Состояние `FOG.EXPLORED` — тайл виден серым, объекты на нём скрыты, но **декорации отображаются с alpha=0.35** (полупрозрачные силуэты деревьев, камней и т.д.).

Флаг `gameMap.fogEnabled = false` отключает туман полностью (удобно для дебага).

### Процедурная генерация и seed

`gameMap.seed` — случайное целое число 0–999999, генерируется при создании `GameMap`. Передаётся во все генераторы:

```js
generateMap(gameMap.seed);       // биомы, высоты, деревни
placeDecorations(gameMap.seed);  // декорации
placeResources(gameMap.seed);    // ресурсы
```

**R (рестарт)** — перегенерация с тем же seed (идентичная карта).
**N (новая карта)** — `gameMap.seed = random()` → рестарт (другая карта).

### HeightMap

`gameMap._heightMap` — float [-1, 1] для каждого тайла. Заполняется `generateMap()` из elevation noise.

- `gameMap.getHeight(ix, iy)` — чтение (0 по умолчанию)
- `gameMap.setHeight(ix, iy, value)` — запись

Применяется в `worldToScreen(wx, wy)`: `y -= tileHeight * HEIGHT_TO_SCREEN` (40 px / ед).

В `Map.draw()` — при перепаде высот > 4 px между (ix,iy) и правым соседом рисуется тёмная трапециевидная боковая грань (`_drawSideFace`).

### Color variation

`gameMap._colorCache` — delta яркости ±0.08 для каждого тайла из detail-noise. Сбрасывается при `setTile()`. Применяется в `Map.draw()` через `_varyColor(baseColor, delta)`.

### Начальные позиции

Перенесены из `constants.js` в `GameMap`:
- `gameMap.initialMinions` — 2 гоблина, относительно `castlePos` (±4 тайла по X)
- `gameMap.castlePos` — позиция замка `{ ix: -30, iy: 0 }`

## 7.1. Процедурная генерация карты

### Модули

| Файл | Экспорт | Описание |
|------|---------|----------|
| `noise.js` | `SimplexNoise`, `fbm`, `createRNG` | Шум и RNG |
| `mapGenerator.js` | `generateMap`, `placeResources`, `placeDecorations`, `lastVillages` | Генерация |
| `VillageGenerator.js` | `generateVillages(gameMap, seed)` | Генерация деревень |
| `decorations.js` | `decorations[]`, `onTileChanged`, `addDecorationsToRenderList`, `getHarvestType`, `harvestDecoration` | Визуальные декорации |

### Слои шума (`generateMap`)

Три независимых `SimplexNoise` с разными seed-смещениями:

| Слой | Scale | Octaves | Назначение |
|------|-------|---------|------------|
| elevation | 0.03 | 4 | Рельеф (горы / равнины / вода) |
| moisture | 0.04 | 3 | Влажность (лес / лёд) |
| detail | 0.08 | 2 | Вариация цвета `_colorCache` |

### Биомы (`getBiome(elevation, moisture)`)

| Условие | Биом |
|---------|------|
| elevation < -0.3 | `water` |
| elevation > 0.6 && moisture > 0.2 | `ice` |
| elevation > 0.5 | `stone` |
| moisture > 0.15 && elevation > -0.1 | `forest` |
| иначе | `plain` |

### Постобработка карты

1. **Расчистка зоны замка** — `_clearZone(castlePos, radius=8)` → все тайлы plain, высоты ≥ 0
2. **Деревни** — `generateVillages(gameMap, seed)` из `VillageGenerator.js` (см. §7.2)
3. **Проверка проходимости** — `_checkPassability()`: BFS от castlePos; если достигнуто < 40% тайлов → `_fixPassability()` удаляет одиночные непроходимые тайлы

### Декорации

`placeDecorations(seed)` расставляет визуальные объекты по биому:

| Биом | Вероятность | Спрайты |
|------|-------------|---------|
| forest | 50% | TREE_1, TREE_2, TREE_3 |
| village | 80% | HOUSE_1, HOUSE_2 |
| village_house | 100% | VILLAGE_HOUSE_S (2/3), VILLAGE_HOUSE_M (1/3) |
| village_road | 50% | VILLAGE_HOUSE_S, SLAB_1 |
| stone | 30% | ROCK_1, ROCK_2 |
| plain | 25% | GRASS_1 (4/5), TREE_3 (1/5) |
| ice | 15% | ICE_CRACK |
| lumber_tile | 100% | TREE_1, TREE_2, TREE_3 |
| mine_tile | 100% | ROCK_1, ROCK_2 |
| farmland | 100% | CROP_GREEN |
| farmland_ripe | 100% | CROP_RIPE |

Смещение (`offset`): village_road — 0.15, lumber_tile — 0.25, mine_tile — 0.2, farmland/farmland_ripe — 0.15.

Каждая декорация: `{ ix, iy, tileIx, tileIy, spriteKey }`. `ix/iy` — точные координаты с рандомным смещением ±0.3. `tileIx/tileIy` — ячейка тайла для fog-check и удаления.

`onTileChanged(ix, iy, oldType, newType, cause)` — при смене тайла через заклинание:
- `oldType === 'wall'` → удалить спрайт стены (WALL_1) с эффектом
- **Камнепад** (`stone → stone`, cause `earth`/`artillery`) → разрушить камни-декорации, спавнить 2–4 камня (typeIndex 1) разлетающихся в 3D (vz 2–5, горизонтальная скорость 1.5–5.0)
- **Лес/лесопилка → plain/rubble** (ветер, земля, артиллерия) → разрушить деревья, дроп 1–2 брёвен (typeIndex 2) подлетающих вверх
- `burning`, `wall`, `water` → удалить все декорации тайла; поля/фермы (если причина не `fire`) дропают пшеницу перед уничтожением; `wall` → добавить спрайт стены (WALL_1)
- `rubble` → разрушение с дропом: `village_house`/`village_square` → камнепад, `mine_tile` → камнепад, `farmland`/`farmland_ripe` → дроп пшеницы, `village_road` → частицы
- `scorched` → удалить деревья, урожай, деревенские постройки (камни остаются)
- `farmland → farmland_ripe` (вода) → заменить CROP_GREEN на CROP_RIPE
- `plain`, `puddle`, `swamp` (если был `forest`) → удалить деревья

**Система спавна предметов**: `setItemSpawnCallback(cb)` в `decorations.js`, подключается в `World.js`. При спавне Item получает `vx/vy/vz` и `state='thrown'`, что запускает полноценную 3D-физику (гравитация, отскоки, затухание).

Декорации добавляются в renderList через `addDecorationsToRenderList(renderList, canvas)` с fog-check, viewport culling и depth = `ix + iy`. Viewport culling использует `camera.zoom` для корректного вычисления видимых границ. Декорации пушатся как data-объекты `{type:'decoration', depth, ox, oy, pixels, scale, explored}` (без замыканий); рендер dispatch в `main.js` обрабатывает `obj.type === 'decoration'` с `ctx.globalAlpha = 0.35` для explored-тайлов.

Колодцы (`VILLAGE_WELL`) размещаются на `village_square` тайлах — один на площадь.

### Визуальные декорации зон производства

~~`Map._drawProductionOverlay()` удалён~~ — заменён спрайтовыми декорациями через `decoByBiome` (см. таблицу выше). Каждый производственный тайл получает декорацию с вероятностью 100%:

| Тайл | Спрайт-декорация |
|------|-----------------|
| `farmland` | CROP_GREEN (зелёный посев, 5×7 px, scale=3) |
| `farmland_ripe` | CROP_RIPE (золотой созревший посев, 5×7 px, scale=3) |
| `mine_tile` | ROCK_1 / ROCK_2 |
| `lumber_tile` | TREE_1 / TREE_2 / TREE_3 |

### Система сбора (harvest)

В `decorations.js` добавлены функции для сбора ресурсов с производственных тайлов:

- `getHarvestType(tileIx, tileIy)` — возвращает `typeIndex` предмета для производственных тайлов: `lumber_tile` → 2 (дерево), `mine_tile` → 1 (камень), `farmland`/`farmland_ripe` → 0 (пшеница). Для прочих тайлов — `null`.
- `harvestDecoration(deco)` — уничтожает декорацию с эффектом `'harvest'`, возвращает `{ typeIndex, ix, iy }` для спавна предмета.
- Эффект `'harvest'` в `destroyDecorationWithEffect` — частицы летят вверх (плавающее рассеивание).

## 7.2. Генерация деревень (`VillageGenerator.js`)

### Вход / выход

```js
generateVillages(gameMap, seed) → Village[]
```

Вызывается из `generateMap()` после расчистки зоны замка, до проверки проходимости. Результат сохраняется в `lastVillages` (экспорт `mapGenerator.js`) и копируется в `World.villages`.

### Алгоритм размещения

1. **Кандидаты** — все тайлы `plain`/`forest`, далеко от замка (≥18 тайлов), не у края карты (≥12 от границы)
2. **Перемешивание** — Fisher-Yates с `createRNG(seed + 7777)`
3. **Выбор 4 позиций** — в порядке: small, small, medium, large; мин. расстояние между деревнями 25 тайлов
4. **Проверка места** — зона `tileSize × tileSize` не содержит `water`/`stone`/`ice`

### Размеры деревень

| Тип | tileSize | Домов |
|-----|----------|-------|
| small | 5×5 | 8–10 |
| medium | 7×7 | 16–20 |
| large | 9×9 | 28–35 |

### Заполнение тайлов

1. Расчистка зоны → `plain`, высоты ≥ 0
2. Центр 2×2 → `village_square`
3. 4 дороги (вверх/вниз/влево/вправо) от центра к краям → `village_road`
4. Случайные позиции (не площадь, не дорога) → `village_house`
5. Оставшиеся `plain` → `village_road` (мощёная площадь)

### Зоны производства

Вокруг каждой деревни в радиусе `half+3..half+8` (ферма) или `half+3..half+12` (шахта, лесоповал):

| Зона | Размер | Требуемый биом | Тайл |
|------|--------|----------------|------|
| Ферма | 3×4 | plain | `farmland` (30% → `farmland_ripe`) |
| Шахта | 2×3 | stone (или plain рядом с stone) | `mine_tile` |
| Лесоповал | 3×3 | forest (или plain рядом с forest) | `lumber_tile` |

### Система производства ресурсов (`ProductionZone.js`)

Каждая зона — экземпляр `ProductionZone`. Создаются в `initWorld()` из `productionZoneTiles` деревень, хранятся в `productionZones[]` (World.js). Быстрый поиск: `findZoneAtTile(ix, iy)` через кэш `_tileToZone` (Map key → zone).

**Параметры:**

| Тип зоны | Период роста (сек) | maxHarvest |
|----------|-------------------|------------|
| farm | 30 | 10 |
| mine | 45 | 10 |
| lumber | 35 | 10 |

**Логика тика (`updateProduction(dt)`):**
1. Подсчёт живых тайлов (не в `DEAD_TILES`: burning, rubble, swamp, water, scorched, steam, puddle)
2. Если все тайлы мертвы → `damaged = true`, производство останавливается
3. `efficiency = alive / total`, `rate = growthRate × boostMult × efficiency`
4. `growthTimer += dt × rate`. При достижении периода → `harvestReady++` (до `maxHarvest`)
5. Синхронизация визуала:
   - Фермы: `_syncFarmVisual()` — синхронизирует количество `farmland_ripe` тайлов с `harvestReady` через `setTile` → `onTileChanged`
   - Шахты/лесоповалы: `_syncResourceVisual()` — добавляет декорации (TREE_1/2/3 или ROCK_1/2) на тайлы зоны при росте `harvestReady`

**Магия и зоны (`handleSpellOnZone` в `tileEffects.js`):**

| Заклинание | Тип зоны | Эффект |
|------------|----------|--------|
| Вода | farm | Буст ×2.0 на 60 сек (ускорение роста) |
| Ветер | farm | Буст ×1.5 на 30 сек |
| Огонь | любая | Тайлы → burning через TILE_TRANSFORMS, зона помечается `damaged` автоматически |
| Земля | любая | Дроп ресурсов через `onTileChanged` (дерево, руда, пшеница) |
| Артиллерия | любая | Усиленный дроп через `onTileChanged` с cause='artillery' |

**Ручной сбор ресурсов зон:** при срыве ресурса рукой → `harvestReady--`, `_lastSyncCount--`:
- Ферма (CROP_RIPE): тайл возвращается в `farmland`, `onTileChanged` ставит CROP_GREEN
- Лесоповал (TREE_1/2/3): декорация удаляется, дерево не отрастает до следующего тика `_syncResourceVisual`
- Шахта (ROCK_1/2): декорация удаляется, камень не отрастает до следующего тика `_syncResourceVisual`

**Инициализация `harvestReady` (`initWorld`):**
- Фермы: подсчёт тайлов `farmland_ripe` (до `placeDecorations`)
- Шахты: подсчёт декораций ROCK_1/ROCK_2 на тайлах зоны (после `placeDecorations`)
- Лесоповалы: подсчёт декораций TREE_1/TREE_2/TREE_3 на тайлах зоны (после `placeDecorations`)

**Вспомогательные поля:** `_tileKeys` (Set координат тайлов зоны для быстрого поиска декораций), `_lastSyncCount` (последнее синхронизированное количество, предотвращает лишние обновления).

**Спрайты зон:** `ZONE_SPRITES = { lumber: ['TREE_1','TREE_2','TREE_3'], mine: ['ROCK_1','ROCK_2'] }`. Тип тайла: `ZONE_TILE_TYPE = { lumber: 'lumber_tile', mine: 'mine_tile' }`.

**Дроп из шахты (`_dropMineOre`):** 70% камень (typeIndex=1), 30% железо (typeIndex=3). 2–4 предмета с 3D-физикой.

### Структура Village

```js
{
    id: 'v0',
    name: 'Дубрава',            // из пула 12 имён
    personality: 'TRD',         // случайный из ['TRD','MIL','REL','COW']

```

### Дебаг-панель [V]

Клавиша **V** / **М** toggle `debugFlags.showVillages`. Панель отображает для каждой деревни:
- Имя, personality, sizeType, pop (houseTiles×2), координаты
- Статус каждой зоны: `type(status, harvestReady/max, next:Ns, boost:Ns)`
  - `status`: `ok` или `DMG` (все тайлы уничтожены)
  - `next:Ns`: секунд до следующего `harvestReady++` (скрыто при `maxHarvest` или `damaged`)
  - `boost:Ns`: оставшееся время буста (только при активном бусте)

### Жители деревень (`Villager.js`, `villageSprites.js`)

Автономные NPC, привязанные к деревням. Наследуют `GameObject` (та же физика подъёма/броска/отскока).

#### Типы жителей

| Тип | HP | Одежда | Роль |
|-----|-----|--------|------|
| `worker` | 30 | Коричневая | Собирает урожай из зон, носит в деревню |
| `carrier` | 30 | Синяя | Переносит грузы между деревнями |
| `militia` | 60 | Серо-зелёная (шлем) | Патрулирует, атакует врагов (dmg=8, cd=1.2с) |
| `refugee` | 15 | Серая рваная | Бежит в ближайшую деревню |

#### Спрайты

Все типы — **6×8 px** (идентичный силуэт MINION_PIXELS гоблина), палитра — человеческая:
- Голова (ряды 0–1): бежевая кожа (`#d4a86a`, `#e0b878`, `#ecc888`)
- Глаза (ряд 2): `#110011`; Рот (ряд 3): `#995555`
- Тело (ряды 4–5): цвет зависит от типа (см. таблицу)
- Ноги (ряды 6–7): тёмно-коричневые (`#5a4020`, `#4a3018`)
- Dead: 8×4 px лежащий спрайт
- Иконки ресурсов: 3×3 px (farm/mine/lumber) — рисуются над головой при переноске

#### Стейт-машина

| Состояние | Описание |
|-----------|----------|
| `idle` | Бродит в радиусе 4 от дома, таймер 3–8 сек |
| `working` | Идёт к зоне производства, собирает урожай |
| `returning` | Несёт ресурс в центр деревни, сдаёт `village.addResource()` |
| `carrying` | Переносит груз (carrier), удаляется по прибытии |
| `fleeing` | Бежит к цели (refugee), скорость ×0.8 |
| `extinguishing` | Тушит пожар: бежит к горящему тайлу (×1.3), ставит `plain` |
| `patrolling` | Militia: обходит радиус 6–10 от центра деревни |
| `fighting` | Militia: преследует и атакует цель |
| `settling` | После приземления: 0.3с → `idle` или `dead` |

Физика (`lifting`, `carried`, `thrown`, `bouncing`, `sliding`) — через `PHYSICS_STATES` и `updatePhysics()`.

#### Рендер при захвате рукой

`draw(hand)` — при `carried`/`lifting` спрайт рисуется под курсором руки через `screenToCanvas(hand.screenX, hand.screenY)` с плавным подъёмом (`lerpT`) и покачиванием (wobble), идентично гоблину `Minion.draw()`.

#### Константы

| Константа | Значение | Описание |
|-----------|----------|----------|
| `VILLAGER_SPEED` | 1.2 | iso units/sec |
| `ARRIVE_DIST` | 0.8 | Порог «дошёл до цели» |
| `DEAD_REMOVE_TIME` | 10 | Сек до удаления трупа |
| `ASSIGN_WORKER_INTERVAL` | 5.0 | Сек между назначениями рабочих |
| `MAX_WORKERS_PER_ZONE` | 3 | Макс. рабочих на 1 зону |

#### Назначение рабочих (`assignWorkers`)

Каждые 5 сек: idle worker'ы назначаются на зоны с `harvestReady > 0`. Зоны сортируются по убыванию `harvestReady`. Максимум 3 рабочих на зону.

#### Экономика деревни

Каждые `VILLAGE_ECON_TICK` (1 сек) для каждой деревни вызывается `updateVillageEconomy()`:

**Ресурсы деревни:**
```js
village.resources = { food, wood, stone }
```
- Стартовая еда: `houseTiles.length * 8`
- `addResource(zoneType, amount)`: farm→food, mine→stone, lumber→wood

**Потребление:** `pop × FOOD_PER_POP_PER_SEC × dt` еды/сек (0.01 = 1 еда на жителя за 100 сек)

**Голод:** при `food = 0` тикает `_starvationTimer`. Каждые `STARVATION_INTERVAL` (10 сек) умирает случайный житель (`_killRandomVillager`). При `pop = 0` → `village.abandoned = true`.

**Рост:** при `food/pop ≥ GROWTH_FOOD_THRESHOLD` (5.0) тикает `_growthTimer`. Каждые `GROWTH_INTERVAL` (60 сек) рождается новый worker (`_spawnVillagerWorker`), `pop++`. Рост ограничен `maxPop = startPop + 10`.

| Константа | Значение | Описание |
|-----------|----------|----------|
| `FOOD_PER_POP_PER_SEC` | 0.01 | 1 еда / 100 сек / жителя |
| `GROWTH_FOOD_THRESHOLD` | 5.0 | Еды/жителя для начала роста |
| `GROWTH_INTERVAL` | 60 сек | Время между рождениями |
| `STARVATION_INTERVAL` | 10 сек | Время между смертями от голода |
| `VILLAGE_ECON_TICK` | 1.0 сек | Частота тика экономики |

**Дебаг-панель [V]** статусы: `OK`, `LOW` (food < pop×2), `STARVING (death:Ns)`, `GROWING (birth:Ns)`, `ABANDONED`. Цвет: красный/оранжевый/зелёный.

#### Интеграция

- **World.js**: `export let villagers = []`, сброс в `initWorld()`/`restartMap()`
- **main.js**: спавн через `setRestartCallback(_spawnVillagers)`, update-loop с передачей `village` и `hand`, рендер в `renderList`, экономический тик `villageEconTimer`
- **input.js**: захват/бросок аналогично миньонам (`hand.grabbedVillager`)
- **Hand.js**: `grabbedVillager` учитывается в `handFree` и wobble-чеках

## 8. Управление

### ЛКМ — физическая рука

| Действие | Функция |
|----------|---------|
| ЛКМ зажать на предмете | Схватить предмет |
| ЛКМ зажать на миньоне | Схватить миньона |
| ЛКМ отпустить | Бросить предмет / миньона (с трекингом скорости) |
| Бросить предмет в замок | Ресурс засчитывается в `castleResources`, предмет исчезает |
| ЛКМ на замке (рука свободна) | Войти в режим стрельбы |
| ЛКМ (режим стрельбы) | Выстрел ядром в прицел |
| ЛКМ (любом месте) | Сбросить выделение `selectedMinions` |

### ПКМ — команды миньонам

| Действие | Функция |
|----------|---------|
| ПКМ клик на скелете | Атаковать: выбранные гоблины → `enterCombat()`; без выделения — 3 ближайших не-воина |
| ПКМ клик на ресурсе | Добывать: выбранные базовые гоблины → `assignGatherTask()`; без выделения — 3 ближайших базовых |
| ПКМ клик на пустом месте | Идти к точке: обычные → `moving_to_point`; воины → кольцевой строй; монахи → переместить тотем; без выделения — 2 ближайших |
| ПКМ перетаскивание (> 5 px) | Лассо-выделение миньонов рамкой (заменяет старое ЛКМ-лассо) |

После команды появляется цветной пульсирующий маркер на 3 секунды:
- **Красный** — атака, **Жёлтый** — добыча, **Зелёный** — движение

Выделение (`selectedMinions`) сбрасывается: ЛКМ в любом месте, через 5 секунд автоматически, или когда все выделенные гоблины погибают.

### Клавиши

**R** — `restartMap()` перегенерирует мир с текущим `gameMap.seed` (идентичная карта). Полный сброс:
- Карта перегенерируется (`generateMap(seed)`)
- Декорации и ресурсы пересоздаются
- Миньоны возвращаются на начальные позиции (wandering)
- Рука сбрасывается (открыта, ничего не захвачено, `selectedMinions = []`)
- Камера сбрасывается (zoom = 1.0)
- Тряска экрана обнуляется
- Статус: «Карта перезапущена!»

## 10. Визуальные эффекты

- **Тени** — эллиптические, уменьшаются и бледнеют с высотой предмета
- **Подсветка наведения** — жёлтая рамка с пульсацией (`sin(time)`)
- **Покачивание** — предмет и рука покачиваются при переносе (sin/cos)
- **Тряска экрана** — при ударе тяжёлых предметов
- **Плавный подъём** — easeOut интерполяция при захвате
- **Подсветка выделения** — зелёная пульсирующая рамка вокруг выделенных миньонов (активна пока `hand.selectedMinions.length > 0`)
- **Маркеры команд** — пульсирующие окружности в точке ПКМ-клика: красный (атака), жёлтый (сбор), зелёный (движение); исчезают за 3 секунды
- **Тряска при уроне** — гоблин резко пошатывается при получении урона (`damageWobble` таймер, 0.4с, sin/cos шейк)
- **Частицы крови** — красные пиксели разлетаются из места удара (8 шт. при уроне, 20 при смерти), с гравитацией и затуханием по alpha
- **Лужица крови** — пиксельный паттерн на полу: при уроне (малая, 2с), при смерти (большая, 4с); рваные края с внешними брызгами; мёртвый гоблин крови не оставляет
- **Тайловые эффекты — трёхфазная система** (`activeTiles[]` в `tileEffects.js`):
  - Каждый `activeTile` имеет `phase: 'impact' | 'active' | 'fading'` + `phaseTimer` (отсчёт внутри фазы)
  - Переходы: `impact → active` по `IMPACT_DUR`; `active → fading` когда `timer >= maxTime - FADING_DUR`
  - **Огонь** (`burning`, 5с → `scorched`):
    - *impact* (0.5с): жёлтый эллипс (`#ffdd44`) затухает по alpha
    - *active*: 3–6 пиксельных языков пламени колышутся вверх через `sin(time + phase)`, мерцание alpha; цвета `#ff4400`–`#ffcc00`
    - *fading* (1.0с): языки убывают в размере, цвета сдвигаются к `#441100`
  - **Пар** (`steam`, 4с → `plain`):
    - *impact* (0.4с): белый эллипс (`#ddddee`) затухает
    - *active*: 4–7 кружков поднимаются вверх с покачиванием, alpha 0.35 × (1 − yOff/30)
    - *fading* (0.8с): alpha × (1 − fadeFrac), радиус кружков увеличивается × 1.6
  - **Лужа** (`puddle`, 10с → `plain`):
    - *impact* (0.4с): голубое кольцо (`#77ddff`) расширяется от 0 до TILE_W/3, alpha убывает
    - *active*: зацикленная рябь-эллипс, скорость расширения × 20 px/с; alpha 0.35 × (1 − ringR/20)
    - *fading* (1.0с): скорость ряби × 0.4, alpha × (1 − fadeFrac)
  - `IMPACT_DUR` и `FADING_DUR` экспортируются из `tileEffects.js` и используются в рендере в `main.js`
- **Разрушение декораций** (`decoParticles[]` в `decorations.js`):
  - При смене тайла (`onTileChanged`) каждый пиксель спрайта декорации превращается в частицу
  - Поведение зависит от `cause`: `fire` — летит вверх (gravity −18), `earth` — щепки в стороны (gravity 130), `artillery` — взрыв во все стороны (speed 70–150, gravity 110), остальное — мягкое рассеивание
  - `cause` передаётся через весь стек: `applySpellToTile(spell, ix, iy, cause)` → `gameMap.setTile(ix, iy, type, cause)` → `onTileChanged(ix, iy, oldType, newType, cause)`
  - `decoParticles` экспортируется через `World.js`; очищается в `initWorld()`; рендерится внутри camera transform после `tileParticles`
- **Частицы заклинаний (`tileParticles[]`)** — массив в `main.js`, camera-space координаты, рендерятся внутри camera transform:
  - **Всплеск воды** (`splash`): 10 пикселей при приземлении water-снаряда; разлетаются с vy вверх, гравитация 150; цвета `#3388cc`–`#77ccff`; живут 0.45–0.55с
  - **Порыв ветра** (`wind`): 14 пикселей при приземлении wind-снаряда; спираль через перпендикулярную компоненту скорости (`angularSpeed`), затухание ×0.97/кадр; цвета `#aaffbb`–`#77ddaa`; 0.6–0.9с
  - **Кольцо ветра** (`windRing`): один expanding эллипс при приземлении wind; без физики, только таймер; от 0 до `WIND_SPELL_RADIUS × TILE_W/2` за 0.35с, stroke `#88ffaa`, alpha убывает
  - **Пыль земли** (`dust`): 8 частиц при приземлении валуна + 2 на каждый пройденный тайл + периодическая пыль за валуном; цвета `#aa8855`–`#665533`; 0.3–0.6с
  - **Обломки валуна**: 14 крупных частиц при столкновении с `stone`/`wall`/замком; цвета `#889999`–`#887744`; 0.4–0.7с; camera shake 8

## 12. Коллизии между предметами

Функция `resolveItemCollisions()` вызывается каждый кадр после обновления физики всех предметов.

- Модель столкновения — упругие круги в изометрической плоскости
- Радиус каждого предмета задан в `ITEM_TYPES[].radius` (iso-единицы)
- Предметы в состоянии `carried` / `lifting` коллизии не участвуют
- Проверка по высоте: столкновение только если `|iz_a - iz_b| < (ra + rb) × 0.5`
- При коллизии: предметы раздвигаются пропорционально массам, обмениваются импульсом
- Коэффициент упругости = `min(bounciness_a, bounciness_b)`
- Неподвижный предмет (`idle` / `settling`) при ударе переходит в `sliding`
- Тряска экрана при сильных ударах (`impactSpeed > 2.0`)

## 13. Миньоны

2 миньона (зелёных гоблина, 6×8 px) на поле. Стейт-машина:

```
Геймплейные состояния (управление игроком):
  free ←────────────────────────────────────────────────────────┐
   │ ПКМ gather                                                  │ нет камней
   ↓                                                             │
  busy ──(подобрал)──→ returning ──(сдал)──→ busy (или free)    │
                                                                 │
  free ←──────────(arrive)──── moving_to_point ←──(ПКМ move)───┘

Боевые состояния (гоблины автоматически):
  free / busy / returning → war ──(в радиусе удара)──→ fighting ──(победа)──→ <savedState>

Разведчик (goblinClass = 'scout'):
  free ←────────────────────────────── free (блуждает по всей карте, стареет 30с)
                                          │ scoutAge >= SCOUT_LIFESPAN
                                          ↓ (natural death)
                                         dead → skeleton (стандартный цикл)

Монах (goblinClass = 'monk'):
  monk_walking ──(достиг тотема)──→ monk_praying ──(тотем сдвинулся > 2 тайла)──→ monk_walking
    ↑ тотем перемещён флагом                    │ пока молится — восстанавливает ману

Физические состояния (рука игрока):
  любое → lifting → carried → thrown → bouncing → sliding → settling → free (или dead)

Смерть и воскрешение:
  settling → dead ──(3 сек)──→ skeleton ──(бросок+приземление)──→ crumbled → (удаление)
```

### Геймплейные состояния

**Зоны патруля** (iso-тайлы от замка `gameMap.castlePos = (-30, 0)`):
| Режим | Зона | Радиус | Поведение |
|-------|------|--------|-----------|
| `free` | 21×21 тайл | ±10 | `pickNewTarget()` выбирает случайную точку в этой зоне |
| `gathererMode` | 42×42 тайл | ±21 | `findNearestResource(items, zoneOnly=true)` ищет ресурсы только в этой зоне |

`gathererMode` определяется **географически** (по позиции ресурса в момент подбора):
- Ресурс в зоне ±21 тайла → `gathererMode = true` → после сдачи продолжает сбор в 42×42
- Ресурс за пределами зоны → `gathererMode = false` → после сдачи становится свободным (21×21)

| # | Состояние | Поведение |
|---|-----------|-----------|
| 1 | `free` | Непрерывно движется к случайной точке в зоне **21×21** вокруг замка; при достижении сразу выбирает новую. Базовое состояние |
| 2 | `moving_to_point` | Движется к `targetX/Y` (точка ПКМ-команды). При достижении (`dist < 0.25`) → `pickNewTarget()`, `free` |
| 3 | `busy` | Идёт к ближайшему ресурсу. Запускается ПКМ-командой «добыча» (`assignGatherTask`). Реагирует на скелетов в радиусе `GOBLIN_AGGRO_RANGE`. Если цель стала недоступна — ищет следующую или уходит в `free` |
| 4 | `returning` | Несёт ресурс к замку (`carriedItem.state = 'goblin_carried'`). Реагирует на скелетов в радиусе `GOBLIN_AGGRO_RANGE`. При достижении замка — сдаёт ресурс (удаляется), ищет следующий → `busy` или `free` |
| 5 | `war` | Видит скелета в радиусе `GOBLIN_AGGRO_RANGE = 10` (воин: `WARRIOR_AGGRO_RANGE = 25`) — движется к нему. Вход из `free`, `busy`, `returning`, `guarding`, `warrior_returning` через `enterCombat()` |
| 6 | `fighting` | В радиусе удара — атакует скелета. Базовый: 10 урона / 1.0с / дальность 0.6. Воин: 20 урона / 1.2с / дальность 0.8. При победе → `exitCombat()` |
| 7 | `guarding` | **Только воин.** Стоит на посту (`guardX/Y`), патрулирует радиус 0.3 тайла. Агрится на скелетов в радиусе 25 тайлов |
| 8 | `warrior_returning` | **Только воин.** Движется к позиции охраны `guardX/Y`. Переход в `guarding` при дистанции < 0.3 |

### Физические состояния (от GameObject)

| Состояние | Поведение |
|-----------|-----------|
| `lifting` / `carried` / `thrown` / `bouncing` / `sliding` | Та же физика, что у предметов (`GameObject.updatePhysics`) |
| `settling` | Пауза 0.3с, затем возврат в `free` или переход в `dead` если `hp <= 0` |
| `dead` | Не двигается; `deadTime` растёт — анимация Труп→Надгробие→Скелет; при `deadTime >= 3.0` → `skeleton` + `isUndead = true`. Используется для угасания тумана войны |
| `skeleton` | Скелет (`isUndead = true`): **враг** — бродит по всей карте, ищет гоблинов в радиусе 15 тайлов, атакует в ближнем бою (15 урона / 1.5с). Скорость 60% от гоблина. Не подчиняется игроку, не выделяется лассо, не раскрывает туман |
| `crumbled` | Конечное состояние скелета после броска; ожидает удаления из `minions[]` |

### ПКМ-выделение лассо

ПКМ-перетаскивание (> 5 px) рисует рамку и заполняет `hand.selectedMinions[]` гоблинами попавшими в область. Выбранные **не меняют состояние** (в отличие от старой системы с флагом).

Выделяются состояния: `free`, `moving_to_point`, `busy`, `returning`, `war`, `fighting`, `guarding`, `warrior_returning`, `monk_walking`, `monk_praying`.

Скелеты (`isUndead = true`) лассо **не выделяются** и не получают задач.

Воины (`goblinClass === 'warrior'`) **выделяются лассо** наравне с обычными гоблинами.

### Система здоровья гоблинов

| Константа | Значение | Описание |
|-----------|----------|----------|
| `MINION_MAX_HP` | 100 | Начальное HP |
| `FALL_DMG_MED_VZ` | 2.5 | Порог vz → средний урон |
| `FALL_DMG_HI_VZ` | 5.0 | Порог vz → большой урон |
| `FALL_DMG_MED` | 25 | Урон при среднем падении (25% HP) |
| `FALL_DMG_HI` | 50 | Урон при сильном падении (50% HP) |

Урон начисляется в хуке `onLand(impactVz)` при приземлении (кроме случая `dead === true`). При `hp <= 0` флаг `dead` устанавливается и после settling гоблин переходит в состояние `dead`.

**Скелет:** HP = 20 (`SKELETON_MAX_HP`). `onLand()` наносит урон от падения аналогично гоблину (25/50). При `hp <= 0` — `pendingBoneEffect`, `pendingRemove = true`, `state = 'crumbled'`.

**Параметры физики:** масса 0.7, отскок 0.25, трение 0.82

**Начальные позиции** (относительно `castlePos = (-30, 0)`):
```
minions[0] @ (castlePos.ix - 4, castlePos.iy) = (-34, 0)
minions[1] @ (castlePos.ix + 4, castlePos.iy) = (-26, 0)
```

**Взаимодействие с рукой:** захват и бросок аналогичны предметам — `hoveredMinion`, `hand.grabbedMinion`. При захвате или переназначении лассо вызывается `dropCarriedItem()` — гоблин бросает камень.

## 13.2. Система скелетов

Когда гоблин погибает (`hp <= 0`), он проходит через три визуальных фазы за 3 секунды, затем воскресает автономным скелетом.

### Анимация смерти (state = `dead`, таймер `deadTime`)

| Диапазон `deadTime` | Визуал | Переход |
|---------------------|--------|---------|
| 0 – 1.0 с | Труп (`MINION_DEAD_PIXELS`) | — |
| 1.0 – 1.4 с | Кросс-фейд труп → надгробие | `α = (t−1.0)/0.4` |
| 1.4 – 2.0 с | Надгробие (`TOMBSTONE_PIXELS`) | — |
| 2.0 – 2.4 с | Кросс-фейд надгробие → скелет | `α = (t−2.0)/0.4` |
| 2.4 – 3.0 с | Скелет (preview) | — |
| ≥ 3.0 с | → `state = 'skeleton'`, `isUndead = true` | автономный режим |

### Скелет как юнит (state = `skeleton`)

- Флаг `isUndead = true` — отличает скелета от гоблина во всех проверках
- **Враг** — ищет и атакует гоблинов (см. «Вражеское поведение» ниже)
- Блуждает по **всей карте** (`gameMap.size`), а не в зоне замка
- Скорость: `MINION_SPEED × 0.6` (`SKELETON_SPEED_FACTOR`)
- HP: `SKELETON_MAX_HP = 20` (получает урон от падения как гоблин)
- **Не** раскрывает туман войны (исключён из `fogSources`)
- **Не** выделяется лассо (`input.js` пропускает `isUndead`)
- **Не** является `hoveredMinion` в состояниях `dead` / `crumbled`
- **Можно** подобрать рукой и бросить — физика идентична гоблину

### Вражеское поведение скелетов

Скелеты — автономные враги, которые преследуют и атакуют гоблинов.

**Алгоритм (каждый кадр в стейте `skeleton`):**
1. Поиск ближайшего живого гоблина (не скелета, не мёртвого, не в руке) в радиусе `SKELETON_AGGRO_RANGE = 15` iso-тайлов
2. Если цель найдена и в радиусе удара (`SKELETON_ATTACK_RANGE = 0.6`) → атака:
   - Урон `SKELETON_ATTACK_DAMAGE = 15` HP
   - Кулдаун `SKELETON_ATTACK_CD = 1.5` секунды
   - Эффект тряски (`damageWobble`) + частицы крови (`pendingBloodEffect`)
   - При `hp <= 0` гоблин умирает → через 3 сек восстаёт скелетом (штатный цикл)
3. Если цель найдена, но далеко → движение к ней (скорость 60% гоблина)
4. Если целей нет → блуждание по карте (`pickNewTarget()` по всей карте)

**Лавинный эффект:** каждый убитый скелетом гоблин через 3 секунды становится новым скелетом, который тоже начинает охотиться — армия скелетов растёт экспоненциально.

| Константа | Значение | Описание |
|-----------|----------|----------|
| `SKELETON_MAX_HP` | 20 | HP скелета (разрушается при 0) |
| `SKELETON_AGGRO_RANGE` | 15.0 | Дистанция обнаружения гоблина (iso-тайлы) |
| `SKELETON_ATTACK_RANGE` | 0.6 | Дистанция удара (ближний бой) |
| `SKELETON_ATTACK_DAMAGE` | 15 | Урон за удар |
| `SKELETON_ATTACK_CD` | 1.5 | Кулдаун между ударами (секунды) |

### Разрушение скелета (`hp <= 0`)

При урон от падения / артиллерии / будущего боя гоблинов:
1. `pendingBoneEffect = { ix, iy }` — маркер для эффекта
2. `pendingRemove = true` — отложенное удаление
3. `state = 'crumbled'` — финальное состояние, рендер пропускается

В `main.js` за кадр: обходим `minions` по `pendingBoneEffect` → спауним 18 частиц в `bloodParticles` с `color` костей (`#d8d0bc` / `#c8c0ac` / `#e0d8c4` / `#b8b0a0`). Затем — обратный проход по `pendingRemove` со сдвигом `hand.grabbedMinion`.

### Спрайты

| Спрайт | Константы | Размер |
|--------|-----------|--------|
| `TOMBSTONE_PIXELS` | `TOMBSTONE_W=6`, `TOMBSTONE_H=9` | 6×9 px — серо-голубой камень с крестом |
| `SKELETON_PIXELS` | `SKELETON_W=6`, `SKELETON_H=8` | 6×8 px — кости с тёмными глазницами |

## 13.1. Система задач гоблинов

Гоблин в состоянии `waiting` может получить задачу от игрока. Задача хранится в поле `task` (null | `'gather'`).

### Задача «Добывать» (`gather`)

**Запуск (ПКМ-команда):**
1. **ПКМ клик на ресурсе** — выделенные базовые гоблины (`goblinClass === 'basic'`) получают `assignGatherTask()`. Без выделения — 3 ближайших базовых гоблина.
2. Воины при этой команде уходят на пост (`warrior_returning`), разведчики → `free`, монахи → `monk_walking`.

**Поток выполнения:**
1. `assignGatherTask(items)` — находит ближайший доступный ресурс через `findNearestResource()` (фильтр по `typeDef.gatherable`), гоблин → `busy`
2. `busy` — гоблин идёт к ресурсу (`targetItem`). Если цель стала недоступна — ищет следующий. При приближении (`dist < 0.6`) и состоянии `idle`/`settling`/`sliding` — подбирает: предмет → `goblin_carried`, гоблин → `returning`
3. `returning` — несёт предмет к замку, предмет следует за гоблином (`iz = 0.5`). При достижении замка (`dist < baseRadius + radius`) — предмет удаляется из `items` через `splice`. Затем: если есть ещё ресурсы → `busy`, иначе → `free`

**Прерывание:** рука подбирает гоблина или лассо переназначает → `dropCarriedItem()` → `task = null`

### Состояние предмета `goblin_carried`

Новое состояние для предметов, которые несёт гоблин:
- Позиция управляется гоблином (обновляется в `returning`)
- Не участвует в коллизиях между предметами (`resolveItemCollisions`)
- Не участвует в коллизии с замком (`Castle.pushObjects`)
- Не доступен для захвата рукой (hover detection в `main.js`)
- Физика (`Item.update`) пропускается

### Поля `Minion` для задачи

| Поле | Тип | Описание |
|------|-----|----------|
| `task` | null \| `'gather'` | Текущая задача |
| `targetItem` | Item \| null | Ссылка на предмет-цель (не индекс!) |
| `carriedItem` | Item \| null | Ссылка на переносимый предмет |
| `gathererMode` | boolean | `true` — ресурс был в зоне ±21, после сдачи ищет следующий в 42×42; `false` — после сдачи уходит в `free` |
| `deadTime` | number | Секунды с момента смерти; используется для плавного угасания зоны видимости в тумане войны |

Объектные ссылки (не индексы) защищают от смещения индексов при `items.splice()`.

## 13.3. Боевая система гоблинов

Гоблины автоматически атакуют скелетов, не требуя команды игрока.

### Проактивный агро (`_tryAggro`)

Метод `_tryAggro(allMinions)` вызывается каждый кадр из состояний `free`, `busy` и `returning`.

**Алгоритм:**
1. Перебирает всех миньонов, пропуская не-скелетов (`!m.isUndead`) и не-активных (`state !== 'skeleton'`)
2. Находит ближайшего скелета в радиусе `GOBLIN_AGGRO_RANGE = 10` iso-тайлов
3. Если нашёл — вызывает `enterCombat(target)` и возвращает `true`

### Вход в бой (`enterCombat`)

При обнаружении врага гоблин:
1. Сохраняет текущий контекст: `savedState`, `savedTask`, `savedTargetItem`, `savedGathererMode`
2. Если нёс ресурс — `dropCarriedItem()` (ресурс падает на землю)
3. Устанавливает `combatTarget = target`, переходит в `state = 'war'`

### Преследование (`war`) и атака (`fighting`)

- `war`: движется к `combatTarget` со скоростью `MINION_SPEED`. При сближении (`dist < GOBLIN_ATTACK_RANGE`) → `fighting`
- `fighting`: атакует каждый `GOBLIN_ATTACK_CD` сек, наносит `GOBLIN_ATTACK_DAMAGE` урона. Отступает в `war` если цель вышла из радиуса удара

**Валидность цели** (`_isValidTarget`): цель жива, не `pendingRemove`, не `crumbled`

### Выход из боя (`exitCombat`)

При победе (цель уничтожена) или потере цели:
1. `combatTarget = null`, `attackCooldown = 0`
2. Восстанавливает состояние по `savedState`:
   - `'gather'` → возобновляет сбор (`busy`, ищет новый ресурс)
   - `'listening'` / `'waiting'` / `'moving'` → возвращается к назначенному игроком состоянию
   - иначе → `free`, `pickNewTarget()`

### Rally-система (призыв на помощь)

Когда гоблин получает урон от скелета, он вызывает союзников:
- Все свободные гоблины (`state = 'free'`) в радиусе `GOBLIN_RALLY_RANGE = 15` iso-тайлов
- Переходят в `war` с тем же `combatTarget`

### Константы боевой системы гоблинов

| Константа | Значение | Описание |
|-----------|----------|----------|
| `GOBLIN_ATTACK_DAMAGE` | 10 | Урон за удар |
| `GOBLIN_ATTACK_CD` | 1.0 с | Кулдаун между ударами |
| `GOBLIN_ATTACK_RANGE` | 0.6 | Дистанция удара (iso-тайлы) |
| `GOBLIN_AGGRO_RANGE` | 10 | Дистанция обнаружения скелетов (iso-тайлы) |
| `GOBLIN_RALLY_RANGE` | 15 | Дистанция призыва союзников при получении урона |

### Конфликт ресурсов (`findNearestResource`)

`findNearestResource(items, zoneOnly, allMinions)` возвращает только предметы в состояниях `idle`/`settling`/`sliding` — то есть ровно те, которые `canPickUp`. Исключает: `carried`, `lifting`, `goblin_carried`, `thrown`, `bouncing`. При `allMinions != null` дополнительно пропускает ресурс, к которому уже движется другой гоблин (`m.state === 'busy' && m.targetItem === item`). Предотвращает «столпотворение» и исчезновение гоблинов при выборе летящего предмета как цели.

## 13.4. Класс воина (`goblinClass = 'warrior'`)

### HUD производства воинов (`main.js`, screen space)

Кликабельный блок под HUD производства гоблинов. Структура аналогична гоблинскому блоку:
- Иконка воина: спрайт `MINION_PIXELS` (scale=2) + шлем `WARRIOR_HELMET_PIXELS` поверх
- Счётчик `воины_в_строю / всего_живых_гоблинов`
- Прогресс-бар золотого цвета (`#cc9922`) — показывает прогресс таймера `warriorUpgradeTimer` при активном производстве
- Кнопка `▶ Запустить` / `⏸ Остановить`

Клик по блоку → `warriorProduction.active = !warriorProduction.active` (перехватывается в capture phase).

`tryUpgradeWarrior()` выполняется только если `warriorProduction.active === true`.

### Апгрейд

Каждые `WARRIOR_UPGRADE_INTERVAL = 8` секунд в `main.js` вызывается `tryUpgradeWarrior()`:
1. Проверяет: `castleResources[3] >= 1` (есть железо)
2. Собирает всех живых базовых гоблинов (`goblinClass = 'basic'`, не мёртв, не скелет, **не в руке** (`state !== 'carried' && state !== 'lifting'`))
3. Выбирает случайного, снимает 1 железо: `castleResources[3]--`
4. Устанавливает `goblinClass = 'warrior'`, получает позицию охраны через `getNextWarriorGuardPos()`
5. Гоблин идёт к посту: `state = 'warrior_returning'`

### Стена воинов (`World.js`)

`getNextWarriorGuardPos()` возвращает следующую позицию охраны вдоль окружности радиуса `WARRIOR_GUARD_RADIUS = 50` тайлов от замка:
```
angle = warriorWall.baseAngle + warriorWall.count * WARRIOR_WALL_STEP
ix = cos(angle) * 50,  iy = sin(angle) * 50
```
- `warriorWall.baseAngle` — случайный угол при старте; сбрасывается при рестарте (`resetWarriorWall()`)
- `WARRIOR_WALL_STEP = 0.04` рад ≈ 2 тайла между соседними воинами на периметре

Воины выстраиваются в линию, образуя «живую стену» вдоль одного направления от замка.

### Поведение воина

| Состояние | Поведение |
|-----------|-----------|
| `warrior_returning` | Движется к `guardX/Y` со скоростью `MINION_SPEED`. Агрится при обнаружении скелетов. При дистанции < 0.3 → `guarding` |
| `guarding` | Стоит на посту. Каждый кадр `_tryAggro()` в радиусе 25 тайлов. При сдвиге > 0.3 — корректирует позицию |

**После боя:** `exitCombat()` для воина всегда → `warrior_returning` (возврат на пост), игнорируя `savedState`.

**После броска рукой:** `onSettle()` → `warrior_returning` (воин не подбирает ресурсы).

### Rally воинов

При нападении скелета на любого гоблина (`skeleton` fighting state):
- Обычные гоблины в `GOBLIN_RALLY_RANGE = 15` тайлов → `enterCombat`
- Воины в `WARRIOR_GUARD_RADIUS = 50` тайлов → `enterCombat` (бегут с любой точки периметра)

### Визуал

Шлем (`WARRIOR_HELMET_PIXELS`, 6×3 px) отрисовывается поверх головы гоблина в `draw()` если `goblinClass === 'warrior' && !isUndead`. Металлические тона: `#7788aa` / `#aabbcc` / `#bbccdd` с бликом `#ccddee`.

Воины **выделяются лассо** и получают команду перемещения в кольцевой строй (см. раздел 13.5).

### Константы воина

| Константа | Значение | Описание |
|-----------|----------|----------|
| `WARRIOR_GUARD_RADIUS` | 50 | Дистанция от замка до поста охраны (iso-тайлы) |
| `WARRIOR_AGGRO_RANGE` | 25 | Дистанция обнаружения врагов |
| `WARRIOR_ATTACK_DAMAGE` | 20 | Урон за удар (в 2× сильнее базового) |
| `WARRIOR_ATTACK_CD` | 1.2 с | Кулдаун между ударами |
| `WARRIOR_ATTACK_RANGE` | 0.8 | Дистанция удара (iso-тайлы) |
| `WARRIOR_WALL_STEP` | 0.04 рад | Угловой шаг между воинами в стене |
| `WARRIOR_UPGRADE_INTERVAL` | 8.0 с | Интервал проверки апгрейда |
| `WARRIOR_IRON_COST` | 1 | Железо на апгрейд |

## 13.6. Класс разведчика (`goblinClass = 'scout'`)

Гоблин-разведчик — временный юнит с расширенной зоной видимости. Создаётся трансформацией любого живого базового гоблина (не в руке), живёт 30 секунд и умирает естественной смертью → превращается в скелет по стандартному циклу.

### HUD производства разведчиков (`main.js`, screen space)

Кликабельный блок под HUD производства воинов. Зелёная рамка (`#33aa44`). Структура:
- Иконка: спрайт `MINION_PIXELS` (scale=2) + капюшон `SCOUT_HOOD_PIXELS` поверх
- Счётчик `разведчиков_живых / SCOUT_MAX_COUNT` (макс 5)
- Прогресс-бар зелёного цвета (`#33aa44`) — показывает таймер `scoutUpgradeTimer` при активном производстве
- Стоимость: 2 дерева + 1 пшеница + базовый гоблин

Клик по блоку → `scoutProduction.active = !scoutProduction.active`.

### Цикл производства

`tryUpgradeScout()` вызывается каждые `SCOUT_UPGRADE_INTERVAL = 10` секунд:
1. Проверяет: `scoutProduction.active`, количество живых разведчиков < `SCOUT_MAX_COUNT = 5`
2. Проверяет ресурсы: `castleResources[2] >= 4` (дерево) и `castleResources[0] >= 1` (пшеница)
3. Собирает всех живых базовых гоблинов (`goblinClass = 'basic'`, не мёртв, не скелет, **не в руке** (`state !== 'carried' && state !== 'lifting'`))
4. Выбирает случайного, снимает 2 дерева и 1 пшеницу
5. Устанавливает `goblinClass = 'scout'`, `scoutAge = 0` — гоблин немедленно начинает блуждать

### Поведение

| Свойство | Значение |
|----------|----------|
| Зона блуждания | Вся карта (`gameMap.size`, как скелеты) — не ограничен 21×21 |
| Зона видимости | `SCOUT_FOG_RADIUS = 5` iso-тайлов (2.5× normal) |
| Жизнь | `SCOUT_LIFESPAN = 30.0` секунд |
| Старение | `scoutAge += dt` только в активных состояниях — **не** в физических (`lifting`, `carried`, `thrown`, `bouncing`, `sliding`) |
| Сбор ресурсов | **Запрещён** — `onSettle()` и `input.js` пропускают разведчика |
| Боевые задачи | Нет — разведчик не агрится на скелетов (нет вызова `_tryAggro`) |
| После смерти | Стандартный `dead → skeleton` цикл; скелет наследует `goblinClass = 'scout'`, но `isUndead = true` исключает его из всей логики разведчика |

### Смерть по старости

```js
// В update(), только активные состояния (не физика):
if (this.goblinClass === 'scout' && !this.dead && !this.isUndead && !physicsStates.includes(this.state)) {
    this.scoutAge += dt;
    if (this.scoutAge >= SCOUT_LIFESPAN) {
        this.dropCarriedItem(); // освобождает ресурс если нёс
        this.dead = true;
        this.state = 'dead';
        this.stateTime = 0;
        this.deadTime = 0;
        return;
    }
}
```

### Исключения из стандартной логики

| Место | Что исключено |
|-------|---------------|
| `Minion.onSettle()` | После приземления не берёт ресурс — `pickNewTarget()`, `state = 'free'` |
| `input.js` — флаг+ресурс | При назначении задачи «добывать» разведчик пропускается |
| `input.js` — флаг+место | При установке флага разведчик пропускается (не переходит в `moving`) |
| `Castle.js` aliveCount | Скелет-бывший-разведчик исключён через `isUndead` |
| Fog sources | Радиус 5 (не 2) если `goblinClass === 'scout'` |

### Визуал

Капюшон `SCOUT_HOOD_PIXELS` (6×3 px, тёмно-зелёный `#3a6030`/`#233d1c`) отрисовывается поверх головы гоблина если `goblinClass === 'scout' && !isUndead`. В панели выделенных юнитов (selection bar) — гоблин с капюшоном.

### Константы

| Константа | Значение | Описание |
|-----------|----------|----------|
| `SCOUT_MAX_COUNT` | 5 | Макс. одновременно живых разведчиков |
| `SCOUT_LIFESPAN` | 30.0 | Секунд до естественной смерти |
| `SCOUT_FOG_RADIUS` | 5.0 | Радиус зрения в iso-тайлах |
| `SCOUT_UPGRADE_INTERVAL` | 10.0 | Секунд между производственными циклами |
| `SCOUT_WOOD_COST` | 2 | Единиц дерева (typeIndex 2) |
| `SCOUT_FOOD_COST` | 1 | Единиц пшеницы (typeIndex 0) |

## 13.7. Класс монаха (`goblinClass = 'monk'`)

Гоблин-молельщик — источник восстановления маны. Создаётся трансформацией любого живого базового гоблина (не в руке). Идёт к тотему и молится, восстанавливая ману хозяина. Управляется через лассо + флаг.

### HUD производства монахов (`main.js`, screen space)

Кликабельный блок под HUD разведчиков. Коричневая рамка (`#443322`). Структура:
- Иконка: спрайт `MINION_PIXELS` (scale=2) + балахон `MONK_ROBE_PIXELS` поверх
- Счётчик `монахов_живых / MONK_MAX_COUNT` (макс 5)
- Прогресс-бар коричневого цвета (`#cc8844`) — показывает таймер `monkUpgradeTimer` при активном производстве
- Стоимость: 1 пшеница + базовый гоблин

Клик по блоку → `monkProduction.active = !monkProduction.active`.

### Цикл производства

`tryUpgradeMonk()` вызывается каждые `MONK_UPGRADE_INTERVAL = 10` секунд:
1. Проверяет: `monkProduction.active`, количество живых монахов < `MONK_MAX_COUNT = 5`
2. Проверяет ресурсы: `castleResources[0] >= 1` (пшеница, typeIndex 0)
3. Собирает всех живых базовых гоблинов (`goblinClass = 'basic'`, не мёртв, не скелет, **не в руке**)
4. Выбирает случайного, снимает 1 пшеницу
5. Устанавливает `goblinClass = 'monk'`
6. Если тотема ещё нет — создаёт тотем в случайном направлении на расстоянии 10–25 тайлов от замка
7. Гоблин получает координаты тотема и идёт к нему: `state = 'monk_walking'`

### Тотем

`monkTotem = { active: false, ix: 0, iy: 0 }` — экспортируемый мутабельный объект в `World.js`.

- Появляется при создании первого монаха; расстояние от замка clamped в `[MONK_TOTEM_MIN_DIST, MONK_TOTEM_MAX_DIST]`
- Рендерится depth-sorted в renderList как пиксельный спрайт `MONK_TOTEM_PIXELS` (4×8 px, масштаб 3)
- Сбрасывается при рестарте карты (`monkTotem.active = false`)

### Поведение монаха

| Состояние | Поведение |
|-----------|-----------|
| `monk_walking` | Движется к `totemX/Y` со скоростью `MINION_SPEED`. При дистанции < 1.0 → `monk_praying` |
| `monk_praying` | Стоит у тотема, покачивается (`sin(performance.now()/700 + idx*1.7) × 2` px). Восстанавливает ману. При отходе тотема > 2 тайлов → `monk_walking` |

### Восстановление маны

Каждый кадр в game loop:
```js
const prayingMonks = minions.filter(m => m.goblinClass==='monk' && m.state==='monk_praying' && !m.dead && !m.isUndead).length;
if (prayingMonks > 0) manaPool.value = Math.min(MANA_MAX, manaPool.value + prayingMonks * MONK_MANA_REGEN * dt);
```

### Управление монахами (лассо + флаг)

Монахи выделяются лассо в состояниях `monk_walking` и `monk_praying`. При выделении переходят в `listening`.

**Флаг → ЛКМ на свободном месте:**
- Тотем перемещается к точке флага (clamped 10–25 тайлов от замка)
- Все живые монахи получают новые `totemX/Y` и переходят в `monk_walking`
- Статус: «Тотем перемещён»

**Флаг → ЛКМ на ресурсе:**
- Монахи пропускаются — возвращаются в `monk_walking` к тотему

### Исключения из стандартной логики

| Место | Что исключено |
|-------|---------------|
| `Minion.onSettle()` | После приземления → `monk_walking` (не подбирает ресурс) |
| `Minion.exitCombat()` | После боя → `monk_walking` |
| `Minion.pickNewTarget()` | Монахи пропускают случайный таргет (нет вызова) |
| `input.js` — флаг+ресурс | Монах пропускается (не добывает) |
| `input.js` — флаг+место | Монах не идёт к флагу — тотем перемещается |

### Визуал

Балахон `MONK_ROBE_PIXELS` (6×3 px, коричневый `#5a3010`/`#3d1e08`) поверх головы гоблина если `goblinClass === 'monk' && !isUndead`. Тотем `MONK_TOTEM_PIXELS` (4×8 px) с золотым навершием и рунным знаком.

### Константы

| Константа | Значение | Описание |
|-----------|----------|----------|
| `MONK_MAX_COUNT` | 5 | Макс. одновременно живых монахов |
| `MONK_MANA_REGEN` | 5.0 | Ед. маны в секунду за одного молящегося монаха |
| `MONK_UPGRADE_INTERVAL` | 10.0 | Секунд между производственными циклами |
| `MONK_FOOD_COST` | 1 | Единиц пшеницы (typeIndex 0) |
| `MONK_TOTEM_MIN_DIST` | 10.0 | Мин. расстояние тотема от замка (iso-тайлы) |
| `MONK_TOTEM_MAX_DIST` | 25.0 | Макс. расстояние тотема от замка (iso-тайлы) |

## 13.5. Ручное управление воинами

Воины выделяются лассо так же, как обычные гоблины, но получают особую команду — перемещение в **кольцевой строй** вокруг целевой точки.

### Выделение

- Рамкой лассо захватываются воины в состояниях `guarding` и `warrior_returning` (добавлены в `SELECTABLE`)
- При выделении воин переходит в `listening` (останавливается), рука берёт флаг
- Выделение воинов и обычных гоблинов может быть смешанным

### Команда перемещения (флаг в руке → ЛКМ на свободном месте)

Воины и обычные гоблины обрабатываются раздельно:

| Тип | Поведение |
|-----|-----------|
| Обычный гоблин | `state = 'moving'`, `targetX/Y = flag.ix/iy` — идут прямо к флагу |
| Воин | Получает уникальную позицию в кольце; `guardX/Y` обновляется, `state = 'warrior_returning'` |

**Кольцевой строй (воины):**
- N воинов равномерно распределяются по окружности вокруг точки флага
- `radius = max(5, N × 10 / 2π)` — шаг дуги между соседними ≈ 10 тайлов
- `angle[i] = 2π × i / N`; позиция: `(flag.ix + radius × cos(angle), flag.iy + radius × sin(angle))`
- При N = 1 воин идёт прямо в точку (radius = 0)
- После прихода воин переходит в `guarding` на новом посту (стандартный механизм `warrior_returning`)

### Команда добычи (флаг в руке + наведено на ресурс)

Воины **не добывают ресурсы**. При выдаче задачи «добывать» через клик на ресурс:
- Обычные гоблины → `assignGatherTask()`
- Воины → сразу `state = 'warrior_returning'` (возвращаются на свой текущий пост охраны)

## 14. Флаг и управление миньонами

Объект `flag` с состояниями: `docked` | `carried` | `placed`.

### Флаг (pixel-art, 5×8 px)
- Древко + жёлтое полотно (`FLAG_PIXELS`)
- Рисуется у курсора с покачиванием, пока `hand.grabbedFlag === true`

### Поток взаимодействия
1. Игрок тянет рамку выделения на пустом поле (от позиции руки)
2. Если попало ≥ 1 миньона — они переходят в `listening`, рука берёт флаг (`hand.grabbedFlag = true`), выделенные подсвечиваются зелёной рамкой
3. **ЛКМ на свободном месте** — флаг устанавливается (`flag.state = 'placed'`), `listening` → `moving` с `targetX/Y = flag.ix/iy`
4. Гоблин достигает флага → `moving` → `waiting` (стоит у флага, ждёт задачу)
5. **Встряхнуть рукой** — флаг рассеивается с эффектом частиц; `listening`/`moving`/`waiting` → `free`

### Подбор поставленного флага

**ЛКМ на поставленном флаге** (рука свободна, `hoveredFlag = true`):
- `flag.state = 'placed'` → `'docked'`; `hand.grabbedFlag = true`
- Все гоблины в `moving`/`waiting` → `listening`; добавляются в `hand.selectedMinions`
- Флаг можно переставить в новое место — гоблины пойдут туда

**Hover-детектирование флага** (`hoveredFlag`):
- Активно только когда рука свободна и `flag.state === 'placed'`
- Радиус: 1.2 iso-единицы от `flag.ix/iy`
- Флаг подсвечивается жёлтой рамкой с пульсацией (уже существующий `drawFlagAt(isHovered)`)
- Приоритет: только если `hoveredItem === null && hoveredMinion === null`

### Мгновенная задача «добывать» (флаг в руке + ресурс)

Когда `hand.grabbedFlag === true`, в `update()` активируется hover-детектирование добываемых ресурсов:
- Ищет ближайший предмет с `typeDef.gatherable === true` в радиусе 1.5 iso-единиц
- При наведении: ресурс подсвечивается жёлтой рамкой, статус — подсказка о клике

**ЛКМ** при `hoveredItem !== null && selectedMinions.length > 0`:
- Все выбранные гоблины в `listening` → `assignGatherTask(items)`
- Флаг рассеивается эффектом частиц у руки (`triggerFlagEffectAtHand`)
- `hand.grabbedFlag = false`, `selectedMinions = []`

### Панель выделенных юнитов (Selection Bar)

Когда `hand.selectedMinions.length > 0` — в нижнем центре экрана отображается панель с иконками выбранных юнитов. Панель видна независимо от флага: после любого лассо-выделения.

**Время жизни выделения:**
- Выделение **сохраняется** до начала новой лассо-операции (новый `mousedown` + drag)
- ЛКМ клик по пустому месту **не сбрасывает** выделение
- Таймер авто-сброса отсутствует

**Отрисовка (screen-space, после ctx.restore()), функция `drawSelectionBar()`:**
- Слоты: `MINION_W × 2 + PAD_X * 2` px в ширину, `MINION_H × 2 + HP_H + 5 + PAD_Y * 2` px в высоту; зазор 4 px
- Панель центрируется горизонтально, отступ 12 px от нижнего края; фон тёмный с рамкой
- Каждый слот: спрайт гоблина в масштабе ×2, поверх — классовый оверлей (шлем воина / капюшон разведчика / балахон монаха), HP-бар внизу слота
- **На hover:** фон `#440000` (горит slot), рамка `#ff6600`, спрайт притушен (alpha 0.5) + символ `×`

**Клик по иконке (capture phase, до input.js):**
- `hand.selectedMinions.splice(selIdx, 1)` — юнит просто убирается из выделения, его состояние не меняется
- `e.stopImmediatePropagation()` — клик не проходит к игровому обработчику

**Блокировка edge scroll:**
- Rect панели (`selectionBarPanel`) обновляется каждый кадр при рендере
- `selectionBarPanel.w = 0` когда панель скрыта — отключает блокировку
- В `update()` в условии `overHud` панель учитывается наряду с правым HUD-панелью

### Детектор встряхивания
- Каждый кадр записывается дельта экранного X руки в `hand.shakeHistory` (кольцевой буфер за 0.6с)
- Подсчитываются смены знака — порог: **5 смен за 0.6с**
- Срабатывает `dismissFlag()`

### Эффект рассеивания флага (`flagEffect`)
- Каждый пиксель `FLAG_PIXELS` превращается в частицу со случайной скоростью
- Гравитация частиц: +120 ед/с²
- Затухание по альфа за `duration = 0.5с` (степенная кривая)
- Координаты хранятся в canvas pre-zoom space (через `screenToCanvas`)

Три точки запуска:
| Функция | Когда вызывается | Источник координат |
|---------|-----------------|-------------------|
| `triggerFlagEffectAtHand()` | Встряхивание, задача через флаг+ресурс | `screenToCanvas(hand.screenX, hand.screenY)` |
| `triggerFlagEffectAtWorld(ix, iy)` | Клавиша 1 с флагом на поле | `worldToScreen(ix, iy)` |
| — (внутри `dismissFlag`) | Любое рассеивание флага в руке | вызывает `triggerFlagEffectAtHand()` |

## 15. Замок

Статичный объект на поле, расположен в iso-позиции `gameMap.castlePos = (-30, 0)`.

### Спрайт
- 10×12 px, масштаб 8 → 80×96 px (ровно 4× высота гоблина 24px)
- Башня с зубцами, ворота, пушка в бойнице
- Пушка (`CASTLE_CANNON_PIXELS`, 4×2 px) рисуется отдельно поверх спрайта с анимацией отдачи

### Рендеринг
- Один entry в renderList с `depth = ix + iy`
- Тень — эллипс под основанием замка
- Рука всегда поверх замка (`depth: Infinity`)

### Коллизия
- Круговая в iso-плоскости, радиус `CASTLE_BASE_RADIUS = 1.3` iso-единиц
- Высота башни `CASTLE_TOWER_HEIGHT = 4.0` — объекты выше не сталкиваются
- Предметы и миньоны в состоянии `carried`/`lifting`/`goblin_carried` коллизию не проходят
- Миньоны в состоянии `returning` коллизию не проходят — чтобы войти в зону замка и сдать камень
- При столкновении объект выталкивается наружу + отражение скорости по нормали (`WALL_BOUNCE`)

| Константа | Значение | Описание |
|-----------|----------|----------|
| `CASTLE_BASE_RADIUS` | 1.3 | Радиус коллизии (iso-единицы) |
| `CASTLE_TOWER_HEIGHT` | 4.0 | Максимальная высота коллизии (world units) |

### Производство гоблинов

Замок производит гоблинов автоматически. Управление через HUD (кликабельный блок под счётчиком ресурсов).

| Константа | Значение | Описание |
|-----------|----------|----------|
| `GOBLIN_MAX` | 25 | Максимум живых гоблинов |
| `GOBLIN_SPAWN_DURATION` | 3.0 с | Время производства одного гоблина |
| `GOBLIN_FOOD_COST` | 1 | Единиц пшеницы за гоблина |
| `GOBLIN_FOOD_TYPE` | 0 | Индекс пшеницы в `ITEM_TYPES` |

**Логика производства (`Castle.update`):**
- **Старт:** если `production.active`, `aliveCount < 25` и есть пшеница → списывается 1 еды, начинается 3-секундный таймер
- **Прогресс:** уже начатый таймер продолжается **независимо** от того, заполнен замок или нет
- **Завершение:** по окончании таймера — если `aliveCount < 25` → `pendingSpawn = true` (гоблин заспавнится); если замок полон — гоблин готов, но не появляется. Еда потрачена в момент старта, не возвращается
- По завершении `pendingSpawn`: `main.js` вызывает `spawnMinion(castle.spawnPoint)` → новый Minion добавляется в `minions[]`
- **Пауза:** если игрок нажимает кнопку при `progress > 0` — текущий гоблин **доделывается** (не прерывается), следующий не начинается

**HUD производства (screen space, под ресурсами):**
- Иконка гоблина (спрайт `MINION_PIXELS`, scale=2)
- Счётчик `живых / 25`
- Прогресс-бар (зелёный = производство идёт; оранжевый = завершает текущего на паузе)
- Статус: `▶ Запустить` / `⏸ Остановить` (лейбл показывает действие по клику, не текущее состояние)
- Клик по блоку → `production.active = !production.active` (перехватывается в capture phase до `input.js`)
- Рамка подсвечивается при наведении (`#aaaaff`), фон светлеет, курсор меняется на `pointer`
- Весь HUD-панель (ресурсы + производство) блокирует edge scroll камеры — `hudPanelRect` проверяется в `update()` перед применением панорамирования
- Отступ от краёв экрана: `HUD_MARGIN = 40px` (≈1 см)

**Спавн:** `World.spawnMinion(ix, iy)` — единственная точка создания новых миньонов вне `initWorld`/`restartMap`. Гоблин появляется у `castle.spawnPoint` (ворота замка: `iy + baseRadius + 0.5`).

### Архитектура (масштабируемость)
- `vx / vy` — готовы для будущего движения замка
- `fireCannon()` / `cannon.recoilTimer` — готовы для будущей системы атак
- `spawnPoint` getter — точка появления миньонов из ворот (`iy + baseRadius + 0.5`)
- `castle` экспортируется как `export let` из `World.js` — переприсваивается при рестарте

## 15.1. Артиллерия замка

Замок может стрелять ядрами по карте. Игрок входит в режим стрельбы, наводит изометрический прицел и делает выстрел по артиллерийской дуге.

### Вход в режим стрельбы

- Рука свободна + ЛКМ на замке (расстояние < `ARTILLERY_GRAB_RADIUS = 2.0` iso)
- Рука фиксируется на замке (`hand.state = 'closed'`), iso-координаты = замок
- Появляется изометрический прицел, следующий за мышью
- Камера управляется edge scroll (мышь у края экрана), зум — клавишами Z/X/C

### Стейт-машина артиллерии (`artilleryMode.state`)

```
aiming → (ЛКМ) → flying → (приземление) → aftermath → (1.5с) → aiming
                                                        ↑
  (Q / Й) → выход из режима в любом состоянии ──────────┘
```

| Состояние | Поведение |
|-----------|-----------|
| `aiming` | Прицел следует за мышью (`screenToIso`). Камера управляется edge scroll (мышь у края экрана). Зум — ручной (Z/X/C) |
| `flying` | Снаряд летит по параболе. Камера следует за ядром (lerp ×4). При `iz ≤ 0` → взрыв |
| `aftermath` | Частицы взрыва обновляются. Через 1.5с камера плавно возвращается к замку (lerp ×3), затем → `aiming` |

### Траектория снаряда

При выстреле рассчитывается время полёта `T = clamp(dist × 0.12, 0.6, 2.5)`:
- `vx = dx / T`, `vy = dy / T` — горизонтальная скорость (линейная)
- `vz = 0.5 × GRAVITY × T` — начальная вертикальная скорость (парабола приземляется точно в цель)
- В полёте: `vz -= GRAVITY × dt`, стандартная физика

### Камера в режиме артиллерии

Формула центрирования: `camera.x = isoToScreen(target).x × zoom`, `camera.y = isoToScreen(target).y × zoom`

| Фаза | Цель камеры | Зум |
|------|-------------|-----|
| `aiming` | Edge scroll (мышь у края экрана) | Ручной (Z/X/C), сохраняется между фазами |
| Момент выстрела | Остаётся на месте | Без изменений |
| `flying` | Снаряд (плавно, lerp ×4) | Без изменений |
| `aftermath` (0–1.5с) | Место взрыва (неподвижна) | Без изменений |
| `aftermath` (>1.5с) | Замок (плавно, lerp ×3) | Без изменений |

Рука в артиллерии: экранная позиция = `isoToScreen(castle) × zoom + canvasCenter - camera`.

### Взрыв

При приземлении ядра (`iz ≤ 0`, `timer > 0.05`):
1. **Тряска экрана** — интенсивность 15
2. **Урон** — все миньоны в радиусе `ARTILLERY_BLAST_RADIUS = 3` iso-тайлов получают 80 HP урона
3. **Отбрасывание** — сила `(1 - dist/radius) × 6`, по направлению от эпицентра; вертикальный импульс `pushForce × 0.5`
4. **Разрушение стен** — все тайлы `wall` в радиусе `ARTILLERY_BLAST_RADIUS` становятся `rubble`
5. **Вырубка леса** — все тайлы `forest` в радиусе взрыва становятся `plain` (cause `'artillery'`); дропается 1–2 ресурса дерева (typeIndex 2) на каждый тайл
6. **Частицы** — 60 огненных + 30 земляных/обломков, гравитация 120, затухание за 0.5–1.1с
7. **Вспышка** — жёлтый эллипс, затухает за 0.15с
8. **Гарь** — тёмный эллипс на земле, появляется через 0.2с, затухает к концу `duration = 1.0`

### Визуал

| Элемент | Описание |
|---------|----------|
| Прицел | Пульсирующий красный эллипс (зона поражения 7×7), пунктирный контур, перекрестие 12px, точка 4×4 |
| Ядро | Спрайт `CANNONBALL_PIXELS` (5×5 px), тень на земле, дымовой след (5 точек) |
| Взрыв | Рваная вспышка (18-точечный полигон `#ffdd44`) + огненные/земляные частицы + рваная гарь на земле (28-точечный полигон `#111100`); точки генерируются при создании взрыва — форма стабильна |
| Пушка замка | Анимация отдачи при выстреле (`castle.fireCannon()`) |

### Константы

| Константа | Значение | Описание |
|-----------|----------|----------|
| `ARTILLERY_GRAB_RADIUS` | 2.0 | Расстояние руки до замка для входа (iso) |
| `ARTILLERY_FLIGHT_TIME_K` | 0.12 | Множитель: T = dist × K |
| `ARTILLERY_MIN_FLIGHT` | 0.6 | Мин. время полёта (секунды) |
| `ARTILLERY_MAX_FLIGHT` | 2.5 | Макс. время полёта (секунды) |
| `ARTILLERY_BLAST_RADIUS` | 3 | Радиус взрыва (iso-тайлы, 7×7) |
| `ARTILLERY_DAMAGE` | 80 | Урон от взрыва |
| `ARTILLERY_RETURN_DELAY` | 1.5 | Пауза после взрыва до возврата камеры |

### Управление

| Действие | Результат |
|----------|-----------|
| ЛКМ на замке (рука свободна) | Вход в режим стрельбы |
| Движение мыши (aiming) | Перемещение прицела; edge scroll у краёв экрана |
| Z/X/C (aiming) | Зум камеры (как в обычном режиме) |
| ЛКМ (aiming) | Выстрел в точку прицела |
| Q / Й | Выход из режима стрельбы |

### Туман войны

Артиллерийский снаряд активно рассеивает туман войны в двух режимах:

| Фаза | Источник тумана | Радиус |
|------|-----------------|--------|
| `flying` | Позиция летящего ядра (обновляется каждый кадр) | 3 тайла |
| После взрыва | Точка эпицентра взрыва (5 секунд) | `ARTILLERY_BLAST_RADIUS + 2` тайла |

**Реализация:**
- `artilleryMode.fogRevealTimer` — таймер, устанавливается в `5.0` при взрыве, убывает каждый кадр
- Пока `state === 'flying'`: `fogSources.push({ ix: projectile.ix, iy: projectile.iy, radius: 3 })`
- Пока `fogRevealTimer > 0`: `fogSources.push({ ix: explosion.ix, iy: explosion.iy, radius: ARTILLERY_BLAST_RADIUS + 2 })`

| Файл | Что добавлено |
|------|--------------|
| `constants.js` | 7 констант `ARTILLERY_*` |
| `sprites.js` | `CANNONBALL_PIXELS` (5×5), `CANNONBALL_W/H` |
| `World.js` | `artilleryMode` — объект состояния + `fogRevealTimer: 0`; сброс в `restartMap()` |
| `input.js` | Захват замка, расчёт траектории, выстрел, сброс камеры, выход по Q |
| `main.js` | `updateArtillery(dt)` (~165 строк) — физика, камера, взрыв, урон; рендер прицела/ядра/взрыва; fog sources для снаряда и зоны взрыва |

## 15.2. Огненный шар (`Fireball`)

### Концепция

Первое боевое заклинание руки. Маг захватывает огненный шар из панели заклинаний и бросает его как обычный предмет. При ударе о землю — взрыв с уроном по площади и горящий патч.

### Взаимодействие с рукой

| Действие | Результат |
|----------|-----------|
| Зажать ЛКМ на панели заклинаний | Рука захватывает шар (`hand.grabbedSpell = 'fireball'`) |
| Отпустить ЛКМ | Бросок — физика как у предмета с `mass 2.5` |
| Приземление (`iz <= 0`) | Взрыв, создаётся `firePatch`, шар уходит в `done` |

### Физика и урон

| Параметр | Значение |
|----------|----------|
| Масса | 2.5 (как камень — тяжёлый бросок) |
| Радиус взрыва | `FIREBALL_BLAST_RADIUS` = 2.5 тайла |
| Урон взрыва | `FIREBALL_BLAST_DAMAGE` = 30 HP |
| Перезарядка | `FIREBALL_COOLDOWN` = 8 с |

### Горящий патч (`firePatch`)

После взрыва в точке падения создаётся объект `firePatch`:

| Параметр | Значение |
|----------|----------|
| Радиус горения | `FIREBALL_BURN_RADIUS` = 2.0 тайла |
| Длительность | `FIREBALL_BURN_DURATION` = 1 с |
| Урон от горения | `FIREBALL_BURN_DPS` = 2 HP/с |
| Туман войны | Зона горения рассеивает туман (`radius = fp.radius`) до конца цикла |

### Визуал

| Элемент | Описание |
|---------|----------|
| Шар в полёте | Пиксельный спрайт 7×7, оранжевое свечение |
| В руке | Рендерится поверх руки, следует за ней |
| Горящий патч | Рваная тёмная гарь (`#111100`, 22-точечный полигон) + 18 пиксельных частиц в огненных цветах (`#ff4400`–`#ffcc00`), поднимающихся вверх; стиль совпадает с артиллерийским взрывом |
| Туман в полёте | Шар рассеивает туман (r=3) пока летит (`thrown`/`bouncing`) |
| Туман от горения | Зона патча (`r=2.0`) остаётся открытой до конца горения |

### Архитектура

- `Fireball extends GameObject` — singleton в `World.js`, сбрасывается через `restartMap()`
- Стейт-машина: `ready → lifting → carried → thrown → bouncing → settling → done`
- `fireball.pendingExplosion = true` — триггер взрыва, обрабатывается в `main.js` следующим кадром
- `firePatches[]` — массив активных горящих зон в `World.js`

### Файлы

| Файл | Что добавлено |

| `constants.js` | `FIREBALL_BLAST_RADIUS/DAMAGE`, `FIREBALL_BURN_RADIUS/DURATION/DPS`, `FIREBALL_COOLDOWN` |

| `World.js` | `fireball` singleton, `firePatches[]`, сброс в `restartMap()` |

| `main.js` | Обработка `pendingExplosion`, урон, создание `firePatch`; обновление патчей (урон по времени, удаление); рендер патча и шара в depth-sort; `fogSources` от `firePatches` |

## 15.5. Бросаемые заклинания (water / earth / wind)

### Концепция

Все три стихийных заклинания теперь бросаются **физическим снарядом** с той же механикой, что и фаербол. Ранее они кастовались мгновенно при отпускании кнопки мыши.

### Реализация

- `SpellProjectile extends GameObject` — один shared-объект в `World.js` для всех трёх заклинаний
- `spellType` (`'water'` | `'earth'` | `'wind'`) устанавливается при захвате из панели, хранится до приземления
- Масса/упругость/трение идентичны фаерболу (`FIREBALL_MASS/BOUNCINESS/FRICTION`)
- `onLand()` — то же что у `Fireball`: выставляет `pendingExplosion`, блокирует последующие отскоки
- Кулдаун **не** хранится в самом объекте — по-прежнему в `spellStates[spellKey]`; выставляется при `pendingExplosion` в `main.js`
- Эффект применяется через `applySpellInRadius(spellKey, ex, ey, radius)` — для water/wind; **earth** использует механику катящегося валуна (см. ниже)
- **Earth spell — катящийся валун**: при приземлении валун начинает **катиться по земле** в направлении броска (`startRolling()`). Линия разрушения вместо кругового взрыва. Трансформирует тайлы на пути (`plain→rubble`, `forest→plain` + дроп дерева, `village→rubble`, `water→swamp`, `burning→scorched`, `puddle→swamp`). Ускоряется на льду (×1.1), вязнет в болоте (×0.3) и воде (×0.5). Останавливается при `speed < BOULDER_MIN_SPEED` или столкновении с `stone`/`wall`/замком. Давит юнитов (40 HP + отбрасывание, кулдаун 0.5с). Пылевой след за валуном.
- **Wind spell — распространение огня**: после knockback-прохода дополнительный цикл ищет горящие тайлы (`burning`) в радиусе ветра; для каждого горящего тайла вычисляется направление от центра взрыва → поджигает через `applySpellToTile('fire', ...)` до 3 соседних тайлов (прямо + ±перпендикуляр); поджигаются `forest`, `plain`, `village`, `farmland`, `farmland_ripe`, `lumber_tile`, `village_house`, `village_road`
- **Wind spell — повал деревьев**: ветер трансформирует `forest → plain` и `lumber_tile → plain`, что вызывает `onTileChanged` с дропом 1–2 брёвен
- Ветер дополнительно отбрасывает юниты в радиусе (`WIND_KNOCKBACK_FORCE`, только при `spellKey === 'wind'`)
- Cooldown не блокирует вылет снаряда, если тот уже был в воздухе от другого заклинания (`spellProjectile.state !== 'ready'` → break)

### Стейт-машина снаряда

```
ready → lifting → carried → thrown → (bouncing/sliding/settling) → done → reset → ready
                                                                ↘ rolling → done (earth only)
```

Переход `done → ready` происходит в `main.js` сразу после установки кулдауна.

### Константы валуна (earth)

| Константа | Значение | Описание |
|-----------|----------|----------|
| `BOULDER_FRICTION` | 0.97 | Трение за 1/60 сек |
| `BOULDER_MIN_SPEED` | 0.3 | Порог остановки (iso ед/сек) |
| `BOULDER_MAX_SPEED` | 6.0 | Макс. скорость при старте |
| `BOULDER_DAMAGE` | 40 | Урон юнитам на пути |
| `BOULDER_DAMAGE_RADIUS` | 0.8 | Радиус проверки урона (iso) |
| `BOULDER_HIT_COOLDOWN` | 0.5 | Секунд между ударами по одному юниту |
| `BOULDER_PUSH_FORCE` | 3.0 | Сила отбрасывания юнитов |

### Файлы

## 15.3. Система маны

### Концепция

Мана — ресурс руки, расходуемый на заклинания. **Пассивной регенерации нет** — мана восстанавливается только молящимися монахами. Независима от кулдауна заклинаний: для броска фаербола нужно **и** достаточно маны **и** завершённый кулдаун.

### Параметры

| Константа | Значение | Описание |
|-----------|----------|----------|
| `MANA_MAX` | 1000 | Максимальный запас маны |
| `MANA_FIREBALL_COST` | 500 | Стоимость огненного шара (50% маны) |
| `MONK_MANA_REGEN` | 5.0 | Восстановление маны за одного молящегося монаха (ед/сек) |

### Реализация

- `manaPool = { value: MANA_MAX }` — экспортируемый мутабельный объект в `World.js` (паттерн как у `screenShake`, `artilleryMode`)
- Регенерация в game loop: подсчёт монахов в `monk_praying` × `MONK_MANA_REGEN * dt`
- Сброс до `MANA_MAX` при рестарте карты (`initWorld()`)
- Проверка при захвате фаербола: `manaPool.value >= MANA_FIREBALL_COST`; при успехе — немедленное списание

### UI — полоса маны

Вертикальная полоса справа от панели заклинаний (screen space):

| Параметр | Значение |
|----------|----------|
| Позиция | `barX = panelX + PANEL_W + 12`, выровнена по высоте панели |
| Размер | 12×(SLOT_SIZE+44) px |
| Заполнение | Снизу вверх, синий градиент `#1a33bb` → `#7799ff` |
| Метка стоимости | Оранжевая горизонтальная черта на уровне 50% (`#ffaa44`) |
| Фон | `#08081a`, прозрачность 0.65 |
| Подпись | «М» над баром |

При недостатке маны (и готовом кулдауне) на слоте фаербола рисуется оверлей «мало маны» (`#000044`, alpha 0.5) + текст `#6688ff`.

## 16.1. Опыт разработки: найденные баги и архитектурные уроки

Этот раздел фиксирует реальные баги, найденные в процессе разработки, и архитектурные выводы из них. Служит справочником при расширении кодовой базы.

---

### Баг 1 — Скелеты блокировали производство гоблинов (критический)

**Файл:** `Castle.js` → `update()`

**Симптом:** После первой смерти гоблина (превращения в скелет) замок переставал производить новых гоблинов, даже если пшеница была.

**Причина:** Фильтр живых гоблинов не исключал скелетов (`isUndead = true`):
```js
// Было (сломано):
const aliveCount = minions.filter(m => m.state !== 'dead').length;
```
Скелеты имеют `state = 'skeleton'` (не `'dead'`), поэтому они засчитывались как живые и занимали слоты.

**Исправление:**
```js
// Стало:
const aliveCount = minions.filter(m => m.state !== 'dead' && !m.isUndead).length;
```

**Архитектурный урок:**
Фильтры «живых» объектов должны учитывать **все** флаги нежизнеспособности (`isUndead`, `pendingRemove`, etc.), а не только проверку `state`. При добавлении нового типа юнита (нежить, призраки) нужно явно решить — занимают ли они слоты в счётчиках. Этот инвариант стоит документировать в комментарии рядом с `aliveCount`.

---

### Баг 2 — Краш при доставке предмета в замок (критический)

**Файл:** `Minion.js` → `update()`, ветка `'returning'`

**Симптом:** Если игрок держал в руке предмет, а гоблин-сборщик одновременно доставлял другой предмет в замок — мог возникнуть краш или рука начинала управлять случайным предметом.

**Причина:** При `splice(idx, 1)` из массива `items[]` код корректировал `hand.grabbedItem` только для случая `> idx`, но не для случая `=== idx`:
```js
// Было (сломано):
if (hand.grabbedItem !== null && hand.grabbedItem > idx) {
    hand.grabbedItem--;
}
// Случай hand.grabbedItem === idx не обрабатывался → dangling index
```

**Исправление:**
```js
// Стало:
if (hand.grabbedItem !== null) {
    if (hand.grabbedItem === idx) {
        hand.grabbedItem = null;          // предмет исчез — рука пуста
    } else if (hand.grabbedItem > idx) {
        hand.grabbedItem--;               // сдвиг индекса после splice
    }
}
```

**Архитектурный урок:**
Паттерн «индекс в массиве» хрупок. При каждом `splice` нужно обрабатывать **три случая**: `< idx` (без изменений), `=== idx` (обнулить), `> idx` (декремент). Более надёжная альтернатива — хранить ссылку на объект, а не индекс: `hand.grabbedItemRef = item` вместо `hand.grabbedItem = 42`. Тогда `splice` не ломает ссылку, и не нужны никакие корректировки.

---

### Баг 3 — Некорректная скорость броска флага (логический)

**Файл:** `input.js` → обработчик `mouseup`, ветка выделения лассо

**Симптом:** После выделения гоблинов лассо флаг "запоминал" скорость предыдущего броска и мог улетать в сторону при следующем отпускании.

**Причина:** При передаче флага в руку через лассо `velocityHistory` не сбрасывался:
```js
// Было (сломано): velocityHistory не очищался
hand.grabbedFlag = true;
hand.state = 'closing';
hand.animProgress = 0;
// velocityHistory = [] — забыли
```

**Исправление:**
```js
hand.grabbedFlag = true;
hand.state = 'closing';
hand.animProgress = 0;
hand.velocityHistory = [];   // ← добавлено
```

**Архитектурный урок:**
Сброс `velocityHistory` — инвариант при любом «захвате» руки. Все три пути захвата (предмет, миньон, флаг) должны его очищать. Это хороший кандидат для метода `hand.grab(target)`, который централизует сброс состояния и исключает подобные пропуски.

---

### Баг 4 — Производительность: `flatMap` в игровом цикле (некритический)

**Файл:** `main.js` → `gameLoop()`, сборка `fogSources`

**Симптом:** Незначительное снижение FPS при большом числе миньонов из-за лишних аллокаций.

**Причина:** `flatMap` создавал новый массив на каждый кадр (60 раз/сек), что увеличивало нагрузку на GC:
```js
// Было:
const fogSources = [
    castleSource,
    ...minions.flatMap(m => { /* возвращает [] или [source] */ }),
];
```

**Исправление:** Заменено на императивный `for...of` с `push`:
```js
// Стало:
const fogSources = [castleSource];
for (const m of minions) {
    if (...) continue;
    fogSources.push({ ix: m.ix, iy: m.iy, radius: 2 });
}
```

**Архитектурный урок:**
В hot path игрового цикла (60+ вызовов/сек) избегать функций высшего порядка (`map`, `filter`, `flatMap`, `reduce`), которые аллоцируют промежуточные массивы. Допустимо использовать их в разовых операциях (инициализация, рестарт), но не в `update()`/`render()`. Профиль: вместо `flatMap` → `push` в loop.

---

### Баг 5 — Мёртвый код в `Hand` (некритический)

**Файл:** `Hand.js` → `constructor()`

**Симптом:** Нет симптома — просто мусор в коде.

**Причина:** Поля `this.targetX` и `this.targetY` объявлялись в конструкторе, устанавливались в нескольких местах, но нигде не читались. Вероятно, остаток от ранней идеи «рука следует к цели».

**Исправление:** Поля удалены из конструктора.

**Архитектурный урок:**
Писать-не-читать поля — признак незавершённой рефакторинга. При добавлении нового поля сразу фиксировать в комментарии, где оно читается. Периодически делать поиск по полю (`grep`) перед тем как оставить его в коде.

---

### Баг 6 — Stale-поля при переходе гоблина в скелета (логический)

**Файл:** `Minion.js` → `update()`, ветка `'dead'`

**Симптом:** Скелет после воскрешения мог немедленно войти в бой, вернуться к несуществующей задаче или двигаться к ресурсу, который уже подобран.

**Причина:** При переходе `dead → skeleton` не очищались поля от прошлой жизни гоблина: `combatTarget`, `savedState`, `task`, `targetItem`, `carriedItem`, `gathererMode`.

**Исправление:** После установки `state = 'skeleton'` принудительно обнуляются все эти поля и вызывается `pickNewTarget()` заново.

**Архитектурный урок:**
Каждый переход в «новую личность» (гоблин → скелет) должен явно объявлять, какие поля принадлежат старой личности. Инициализация через конструктор не помогает для юнитов, меняющих тип в рантайме — нужен явный reset-блок при каждом таком переходе.

---

### Баг 7 — Несколько гоблинов идут к одному ресурсу (логический)

**Файл:** `Minion.js` → `findNearestResource()`

### Баг 10 — scoutAge росло пока гоблин в руке (логический)

**Файл:** `Minion.js` → `update()`, счётчик возраста разведчика

**Симптом:** Разведчик мог умереть пока игрок держит его в руке — труп оказывался в руке.

**Причина:** Условие старения не проверяло физические состояния:
```js
// Было (сломано):
if (this.goblinClass === 'scout' && !this.dead && !this.isUndead) {
    this.scoutAge += dt; // росло даже в 'carried'
```

**Исправление:**
```js
// Стало:
if (this.goblinClass === 'scout' && !this.dead && !this.isUndead
    && !physicsStates.includes(this.state)) {
    this.scoutAge += dt;
```

**Архитектурный урок:**
Таймеры «жизни» юнита должны учитывать физические состояния — пока юнит в руке игрока, он «заморожен». Проверка `physicsStates.includes(state)` — готовый паттерн для любого подобного таймера.

### Баг 11 — Ресурс зависал при смерти разведчика по старости (логический)

```js
this.dropCarriedItem(); // до установки dead = true
```

Любой путь смерти (урон, старость, будущие эффекты) должен вызывать `dropCarriedItem()`. Хороший кандидат для метода `die()`, централизующего все инварианты смерти.

```js
if (this.goblinClass === 'scout') {
```

### Баг 13 — Мёртвые миньоны оставались в selectedMinions (логический)

**Файл:** `main.js` → `update()`, управление `hand.selectedMinions`

**Симптом:** Если разведчик (или любой гоблин) умирал пока был выделен, его индекс оставался в `hand.selectedMinions`. Клик по иконке в selection bar выполнял `m.state = 'free'` на мёртвом миньоне.

**Причина:** Очистка `selectedMinions` происходила только при явном действии игрока (снятие флага, клик по иконке), но не при смерти миньона.

**Исправление:** После цикла обновления миньонов — фильтрация мёртвых/нежити из `selectedMinions`. При пустом списке — автоматическое снятие флага:

```js
if (hand.grabbedFlag && hand.selectedMinions.length > 0) {

        const m = minions[idx];
        return m && !m.dead && !m.isUndead;

    if (hand.selectedMinions.length === 0) {
        flag.state = 'docked';
        hand.grabbedFlag = false;

    }
}
```

**Архитектурный урок:**
`selectedMinions` — производное состояние от `minions[]`. Его нужно синхронизировать при каждом изменении `minions[]`: смерть, splice (уже есть), смена класса. Альтернатива — хранить ссылки на объекты, а не индексы, и фильтровать по `m.dead || m.isUndead`.

### Баг 14 — Огненный шар вылетал выше ладони (визуальный)

**Файл:** `input.js` → обработчик `mouseup`, блок `hand.grabbedSpell === 'fireball'`

**Симптом:** При броске огненного шара он мгновенно оказывался выше руки и уходил ещё дальше вверх до начала падения.

**Причина:** В момент броска `fireball.iz` оставалось равным `CARRY_HEIGHT = 1.5` (высота переноски). В состоянии `thrown` физика применяет `vz` поверх текущего `iz`, поэтому шар стартовал с высоты 1.5 и сразу летел ещё выше от положительного `vz`.

Для сравнения — бросок предмета явно сбрасывает высоту перед стартом:
```js
item.iz = 0;  // ← есть у предметов
```
У огненного шара этой строки не было.

**Исправление:** Добавлена одна строка в `input.js`:
```js
fireball.iz = 0;
fireball.vx = throwVel.vx;
fireball.vy = throwVel.vy;
fireball.vz = throwVel.vz;
```

**Архитектурный урок:**
Любой новый тип бросаемого объекта должен сбрасывать `iz = 0` перед переходом в `thrown`. Это паттерн всех предметов и гоблинов — он обязателен и при добавлении нового объекта легко пропускается, если нет централизованного метода `startThrow()`.

### Баг 15 — Переданный предмет застревал в `goblin_carried` при смене задачи (логический)

**Файл:** `Minion.js` → `assignGatherTask()`

**Симптом:** Если гоблин нёс предмет в замок (`returning`), а игрок кликал ПКМ на другом ресурсе — гоблин бросал текущий предмет, но с этим брошенным предметом больше нельзя было взаимодействовать: рука его не захватывала, другие гоблины игнорировали.

**Причина:** `assignGatherTask` просто присваивал `this.carriedItem = null`, не меняя состояние самого предмета. Предмет так и оставался `goblin_carried` навсегда — без хозяина, без возможности выйти из этого состояния.

**Исправление:** Перед сменой задачи явно переводить переносимый предмет в состояние `thrown` (с нулевыми скоростями) — физика его опустит на землю и предмет станет доступен снова.

### Баг 16 — Гоблин исчезал при смене задачи во время переноски (критический)

**Файл:** `Minion.js` → `assignGatherTask()`, `findNearestResource()`, `update()` case `busy`

**Симптом:** Если гоблин нёс ресурс в замок (`returning`), а игрок кликал ПКМ на другом ресурсе — гоблин мгновенно исчезал с карты.

**Причина:** Цепочка из трёх проблем:
1. `assignGatherTask` роняет предмет A в `thrown` прямо на позиции гоблина (там он находился во время переноски).
2. `findNearestResource` **не фильтровала** предметы в состоянии `thrown`/`bouncing` — предмет A оказывался «ближайшим ресурсом» (dist ≈ 0) и назначался новой целью.
3. В состоянии `busy` при `dist < 0.6` && `canPickUp = false` (предмет ещё летит) код пытался двигаться по формуле `dx / dist` при `dist ≈ 0` → **NaN-координаты** → гоблин рисовался за пределами экрана.

**Исправление:**
- `findNearestResource`: добавить `thrown` и `bouncing` в список исключаемых состояний.
- `busy` → проверка валидности цели: также инвалидировать `thrown`/`bouncing` (перейти к поиску следующей цели).
- `busy` → движение: добавить `dist > 0.001` перед делением — страховка от деления на ~0.

**Архитектурный урок:**
`findNearestResource` должна возвращать только предметы в состояниях `idle`/`settling`/`sliding` — то есть ровно те, которые `canPickUp`. Расширение фильтра «только не-в-руке» недостаточно: летящие предметы нестабильны и могут оказаться на нулевом расстоянии.

```js
// Было:
import { decorations } from './decorations.js';

import { decorations } from './decorations.js?v=3';
```

### Баг 18 — Гоблины-сборщики не могли добывать ресурсы вблизи замка (критический)

**Файл:** `Minion.js` → `findNearestResource()`, `onSettle()`, ветка `busy`

**Симптом:** Гоблины, назначенные флагом на сбор, немедленно переходили в `free` — не находили ресурсов в зоне. Ресурсы у замка лежали, но были «невидимы» для сборщиков.

**Причина:** Зона сбора `GATHER_ZONE_RADIUS = 21` проверялась по **абсолютным** iso-координатам:
```js
// Было (сломано):
if (Math.abs(item.ix) > GATHER_ZONE_RADIUS || Math.abs(item.iy) > GATHER_ZONE_RADIUS) continue;
```
Замок находится в `(-30, 0)`. Ресурс у замка имеет `ix ≈ -27`. `Math.abs(-27) = 27 > 21` → ресурс всегда вне зоны, несмотря на то что он в 3 тайлах от замка.

**Исправление:** (3 места в `Minion.js`)
```js
// Стало — координаты относительно castlePos:
if (Math.abs(item.ix - gameMap.castlePos.ix) > GATHER_ZONE_RADIUS || Math.abs(item.iy - gameMap.castlePos.iy) > GATHER_ZONE_RADIUS) continue;
```

**Архитектурный урок:**
Все радиусные проверки от замка должны вычисляться **относительно `gameMap.castlePos`**, а не абсолютного `(0, 0)`. Комментарий «42×42 вокруг замка» должен был подсказать это — замок не в центре карты.

### Баг 19 — Сплит модулей: разные версии URL → раздельное состояние (системный, критический)

**Файлы:** `input.js`, `tileEffects.js`, `Castle.js`, `Fireball.js`, `Hand.js`, `Minion.js`, `SpellProjectile.js`, `constants.js`

**Симптом (множественный):**
- Рука не могла захватывать гоблинов (`minions[hoveredMinion]` → `undefined`)
- P/З не отключала туман войны (`debugFlags` не менялся)
- Эффекты заклинаний не работали (`activeTiles` оставался пустым после броска)
- Магия не летела / не давала эффектов

**Причина:** Python SimpleHTTP отдаёт `World.js` и `World.js?v=3` как один и тот же файл, но браузер кэширует ES-модули **по URL-строке**. Два разных URL = два независимых экземпляра модуля с двумя независимыми массивами/объектами:

| URL в импорте | Файлы | Экземпляр |
|---|---|---|
| `./World.js?v=3` | `main.js` | Инст. A — заполняется `initWorld()` |
| `./World.js` | `input.js`, `tileEffects.js` | Инст. B — всегда пустой |

`main.js` рендерил из инст. A, `input.js` читал из инст. B → `minions[i]` = `undefined`.

**Исправление:** Унифицировать URL во **всех** импортах одного модуля:
```js
// input.js, tileEffects.js:
from './World.js'  →  from './World.js?v=3'

// Minion.js, input.js:
from './tileEffects.js'  →  from './tileEffects.js?v=2'

// Castle.js, Fireball.js, Hand.js, Minion.js, SpellProjectile.js, constants.js, input.js:
from './sprites.js'  →  from './sprites.js?v=3'
```

**Архитектурный урок:**
При добавлении `?v=N` к импорту в одном файле — немедленно обновить **все** остальные импортёры того же модуля. Несогласованные версии хуже чем отсутствие версий: они создают скрытый сплит состояния, который ломает всё что зависит от разделяемых массивов-синглтонов (`minions`, `items`, `activeTiles`, `debugFlags` и т.д.). Правило: один модуль = один URL во всём проекте.

### Баг 20 — Замок рендерился ниже поверхности земли (визуальный)

**Файл:** `Castle.js` → локальная функция `worldToScreen()`

**Симптом:** Замок «тонул» в рельефе — его основание оказывалось ниже видимой поверхности тайлов на слегка приподнятом terrain.

**Причина:** `Castle.js` имел собственную копию `worldToScreen`, которая **игнорировала высоту тайла**:
```js
// Было (локальная копия, без высоты):
function worldToScreen(wx, wy) {
    const iso = isoToScreen(wx, wy);
    return { x: iso.x + canvas.width/2, y: iso.y + canvas.height/2 - CAMERA_OFFSET_Y };
}
```
`isometry.js` правильно вычитает `tileHeight * HEIGHT_TO_SCREEN`, но локальная копия этого не делала.

**Исправление:**
```js
// Стало — импорт из isometry.js вместо локальной копии:
import { getDepth, worldToScreen } from './isometry.js';
```

**Архитектурный урок:**
`worldToScreen` — **единственная каноническая реализация** в `isometry.js`. Никаких локальных копий в других файлах. При добавлении нового визуального объекта всегда импортировать `worldToScreen` из `isometry.js`, а не переписывать заново.
