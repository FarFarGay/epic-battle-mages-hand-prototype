# Спецификация: Прототип Руки — Epic Battle Mages

## 1. Обзор

**Проект:** Прототип механики руки для игры "Epic Battle Mages"
**Технологии:** Vanilla HTML5 Canvas 2D, JavaScript ES Modules (без зависимостей, без сборщика)
**Файл:** `index.html` + модули в `js/` (~2100 строк суммарно)
**Запуск:** Только через локальный HTTP-сервер (ES modules требуют HTTP): `python3 -m http.server 8080`

Прототип реализует основную механику игры — руку волшебника, которая служит курсором. Рука может подбирать, переносить и бросать предметы с реалистичной физикой (гравитация, отскоки, трение, скольжение). Также реализовано управление миньонами через флаг.

## 2. Архитектура

### Файловая структура (ES Modules)

```
index.html              — HTML-оболочка (27 строк), загружает js/main.js
js/
├── constants.js        — физические константы, ITEM_TYPES (без начальных позиций)
├── sprites.js          — пиксельные спрайты (*_PIXELS массивы), без логики
├── isometry.js         — изометрическая проекция, camera, screenToIso/isoToScreen
├── renderer.js         — canvas/ctx, drawPixel, drawPixelArt, drawIsoDiamond, тени
├── Map.js              — class GameMap: размер, типы тайлов, туман войны, начальные позиции
├── GameObject.js       — базовый класс: общая физика (lifting→carried→thrown→…)
├── Item.js             — class Item extends GameObject: логика и отрисовка предмета
├── Minion.js           — class Minion extends GameObject: AI, HP, смерть, отрисовка
├── Hand.js             — class Hand: движение руки, захват, бросок, shake-детектор
├── Castle.js           — class Castle: коллизия, рендер, пушка с отдачей, spawnPoint
├── World.js            — items[], minions[], flag, castle, screenShake, resolveCollisions, restartMap
├── input.js            — все обработчики событий мыши и клавиатуры
└── main.js             — игровой цикл: update(), render(), gameLoop()
```

### Иерархия классов

```
GameObject (базовая физика)
├── Item     (ресурсы: пшеница, камень, дерево, железо, свиток)
└── Minion   (гоблины: AI блуждания, HP, мёртвое состояние → скелет)

Hand         (рука-курсор, не наследует GameObject)
Castle       (замок: коллизия, рендер, пушка — не наследует GameObject)
```

**`GameObject`** содержит метод `updatePhysics(dt, hand, triggerShake)` — общую стейт-машину физики для состояний `lifting`, `carried`, `thrown`, `bouncing`, `sliding`. Хуки для переопределения в подклассах:
- `onLand(impactVz)` — при приземлении (используется в `Minion` для урона от падения)
- `onSettle()` — после таймаута settling (переход в `free` или `dead`)

### Рендеринг
- **Изометрическая проекция** — угол 45° (`ISO_ANGLE = π/4`)
- **Тайловая сетка** — `TILE_W = 64px`, `TILE_H ≈ 45px`
- **Pixel-art** — масштаб `PIXEL_SCALE = 3`, сглаживание отключено (`imageSmoothingEnabled = false`)
- **Глубинная сортировка** — по сумме `ix + iy` для корректного порядка отрисовки
- **Рука** — всегда рендерится последней (`depth: Infinity`), поверх всех объектов

### Game Loop
- `requestAnimationFrame` с вычислением `dt` (delta time)
- Ограничение `dt` до 0.05с (20 FPS минимум) для стабильности физики
- Цикл: `update(dt)` → `render()`

### Стейт-машины
- **Рука:** `open` → `closing` → `closed` → `opening` → `open`
- **Предметы:** `idle` → `lifting` → `carried` → `thrown` → `bouncing` → `sliding` → `settling` → `idle`
- **Миньоны (геймплей):** `free` → `listening` → `moving` → `waiting`; флаг рассеян → `free`
- **Миньоны (физика):** любое → `lifting` → `carried` → `thrown` → `bouncing` → `sliding` → `settling` → `free` (или `dead`)

## 3. Механика руки

### Движение
- Рука следует за курсором мыши с плавной интерполяцией (`lerp = 1 - 0.001^dt`)
- Скрытый системный курсор (`cursor: none`), заменён красной точкой 3×3px

### Спрайты
- **Открытая ладонь** (`HAND_OPEN_PIXELS`) — 10×12 px, вид сверху-сбоку
- **Закрытая ладонь** (`HAND_CLOSED_PIXELS`) — пальцы загнуты внутрь
- Анимация перехода: покадровая интерполяция при `closing`/`opening`
- Покачивание при захвате (sin/cos колебания)

### Захват и бросок
1. **ЛКМ зажать** — если есть предмет в радиусе 1.5 iso-единиц → захват
2. Предмет плавно поднимается к руке (состояние `lifting`)
3. Предмет следует за рукой (состояние `carried`)
4. **ЛКМ отпустить** — бросок с физикой
5. Можно ловить предметы в воздухе (если высота `iz < 2.0`)

### Трекинг скорости
- Хранение последних 10 кадров скорости руки (`velocityHistory`)
- Взвешенное усреднение (последние кадры важнее)
- Учёт массы предмета при расчёте скорости броска

## 4. Физическая система

### Константы

| Константа | Значение | Описание |
|-----------|----------|----------|
| `GRAVITY` | 8.0 | Ускорение свободного падения (ед/с²) |
| `CARRY_HEIGHT` | 1.5 | Высота переноса предмета |
| `LIFT_SPEED` | 4.0 | Скорость подъёма предмета |
| `THROW_VZ_BASE` | 3.0 | Базовый вертикальный импульс при броске |
| `THROW_SCALE` | 1.5 | Множитель скорости руки → скорость броска |
| `BOUNCE_MIN_VZ` | 0.3 | Минимальная вертикальная скорость для отскока |
| `MAX_BOUNCES` | 5 | Максимальное количество отскоков |
| `SLIDE_STOP` | 0.05 | Порог остановки скольжения |
| `HEIGHT_TO_SCREEN` | 40 | Пикселей на единицу высоты |
| `VELOCITY_HISTORY` | 10 | Кадров для усреднения скорости |
| `MAX_THROW_SPEED` | 8.0 | Максимальная скорость броска (iso ед/с) |
| `WALL_BOUNCE` | 0.6 | Коэффициент отскока от стен |
| `CAMERA_OFFSET_Y` | 40 | Вертикальное смещение центра мира (px) |
| `AIR_RESISTANCE` | 0.99 | Коэффициент сопротивления воздуха (за 1/60 сек) |
| `MINION_SPEED` | 1.5 | Скорость блуждания миньона (iso ед/с) |

### Стейт-машина предметов

```
idle → lifting → carried → thrown → bouncing → sliding → settling → idle
                                ↘ sliding → settling → idle
```

| Состояние | Поведение |
|-----------|-----------|
| `idle` | На земле, `iz=0`, можно захватить |
| `lifting` | Плавный подъём к `CARRY_HEIGHT`, easeOut анимация, движение к руке |
| `carried` | Следует за рукой, `iz = CARRY_HEIGHT` |
| `thrown` | Гравитация тянет вниз, горизонтальное движение, сопротивление воздуха (`Math.pow(AIR_RESISTANCE, dt * 60)`) |
| `bouncing` | Отскок: `vz` инвертируется × `bounciness`, горизонтальная скорость × `friction` |
| `sliding` | На земле, экспоненциальное трение замедляет `vx/vy` |
| `settling` | Пауза 0.15с перед возвратом в `idle` |

### Тряска экрана
- При первом ударе предмета о землю
- Интенсивность = `mass × |vz| × 0.5`
- Экспоненциальное затухание
- Максимальная интенсивность: 15px

| 3 | Железо  | 3.0 | 0.2 | 0.80 | 0.40 | Снаряжение гоблинов; самый тяжёлый ресурс |

## 6. Камера

### Зум
- **Z** (или **Я**) — отдалить на 10% (×0.9), минимум 0.2×
- **X** (или **Ч**) — приблизить на 10% (×1.1), максимум 3.0×
- **C** (или **С**) — сброс зума к 1.0×
- Плавная интерполяция (`dt × 8`)

### Панорамирование (Edge Scroll)
- Камера автоматически движется когда рука подходит к краю экрана
- Зона срабатывания: 12% ширины/высоты экрана с каждого края (`EDGE_ZONE = 0.12`)
- Максимальная скорость: 600 px/с в screen space (`PAN_SPEED = 600`)
- Линейный спад скорости: чем глубже рука в зону — тем быстрее движение
- `camera.x/y` хранят смещение в screen-пикселях
- Все функции `screenToCanvas` во всех модулях учитывают `camera.x/y` — объекты в руке всегда следуют за курсором

### Рендеринг камеры
- Зум через `ctx.scale()` относительно центра экрана
- Панорамирование через `ctx.translate(-camera.x, -camera.y)` перед зумом
- Тряска через `ctx.translate()` поверх зума
- Вертикальное смещение центра мира задаётся константой `CAMERA_OFFSET_Y = 40` (используется в `screenToIso` и `worldToScreen`)
- Индикатор зума в левом нижнем углу (показывается при отклонении от 100%)

## 7. Карта (`Map.js`)

Класс `GameMap` — единственный источник истины о размере поля, тайлах, тумане войны и начальных позициях. Синглтон `gameMap` импортируется в `World.js` и `main.js`.

### Размер и границы

- `gameMap.size = 80` → поле 161×161 тайл (от -80 до +80 по обеим осям)
- Изменение в рантайме: `gameMap.resize(newSize)` — очищает данные вне новых границ
- Невидимые стены по краям поля (`size + 0.5`), предметы отскакивают с `WALL_BOUNCE = 0.6`

### Типы тайлов

Каждый тайл хранит ключ типа (строка). Доступные типы в `TILE_TYPES`:

| Ключ | Цветовая гамма |
|------|----------------|
| `plain` | Тёмно-фиолетовый (по умолчанию) |
| `forest` | Тёмно-зелёный |
| `water` | Тёмно-синий |
| `stone` | Тёмно-серый |
| `village` | Коричневый |
| `ice` | Холодный синий |

API: `gameMap.setTile(ix, iy, 'forest')` / `gameMap.getTile(ix, iy)`

### Туман войны

Состояния тайлов (`FOG`): `hidden` → `explored` → `visible`

| Состояние | Визуал |
|-----------|--------|
| `hidden` | Чёрный тайл (`#0a0a12`) |
| `explored` | Тёмно-серый (посещено, но вне обзора) |
| `visible` | Нормальный цвет типа тайла |

Туман обновляется каждый кадр через `gameMap.tickFog(sources)`:
- Все `visible` → `explored`
- Затем `revealAround(ix, iy, radius)` для каждого источника

**Источники видимости** (радиус в тайлах):
| Источник | Радиус | Форма | Условие |
|----------|--------|-------|---------|
| Замок | 10 (21×21 тайл) | `square` | всегда |
| Живой миньон (basic/warrior) | 2 | `circle` | `state !== 'carried' && state !== 'lifting'` |
| Живой разведчик | 5 | `circle` | `goblinClass === 'scout'`, те же условия |
| Мёртвый миньон | 2 → 0 за 10с | `circle` | `deadTime < 10` — плавное угасание |

**Правила тумана для миньонов:**
- Рука **собственной видимостью не обладает** — поднять предмет вне зоны видимости невозможно.
- Гоблин **в руке** (`carried` / `lifting`) туман **не раскрывает** — водить гоблином по карте как разведчиком нельзя.
- После **смерти** гоблина его зона видимости плавно сужается от 2 до 0 за **10 секунд** (поле `deadTime` растёт в state `dead`).

**Ограничения руки вне зоны видимости:**
- Рука **не может поднимать** предметы или миньонов вне `FOG.VISIBLE` (hover detection пропускает невидимые тайлы).
- Если рука **выходит за пределы** `FOG.VISIBLE` с уже захваченным предметом или миньоном — она **немедленно роняет** его с нулевой скоростью (свободное падение на месте). Это предотвращает разведку картой через перенос гоблина.
- **Исключение — брошенный гоблин:** если гоблин **заброшен в темноту** и выжил при падении, он становится дружественным источником видимости (радиус 2). В его зоне видимости рука **может** подобрать как самого гоблина, так и любые предметы рядом с ним. Это намеренная игровая механика — бросок гоблина как «разведывательный» риск.

Объекты вне `FOG.VISIBLE` скрыты и недоступны для взаимодействия (hover, захват). Состояние `FOG.EXPLORED` — тайл виден серым, но объекты на нём скрыты.

**Исключение — флаг:** установленный флаг (`flag.state === 'placed'`) всегда отрисовывается поверх тумана. Игрок должен видеть куда поставил флаг даже за пределами зоны видимости базы.

Флаг `gameMap.fogEnabled = false` отключает туман полностью (удобно для дебага).

### Начальные позиции

Перенесены из `constants.js` в `GameMap`:
- `gameMap.initialItems` — 7 предметов
- `gameMap.initialMinions` — 2 гоблина
- `gameMap.castlePos` — позиция замка `{ ix: 0, iy: 0 }` (центр карты)

## 8. Управление

| Клавиша / Действие | RU раскладка | Функция |
|---------------------|--------------|---------|
| ЛКМ зажать | — | Схватить предмет / миньона / флаг с земли |
| ЛКМ отпустить | — | Бросить предмет / миньона |
| Тяни на пустом месте | — | Выделить миньонов рамкой |
| ЛКМ (флаг в руке, на свободном месте) | — | Установить флаг; выделенные гоблины идут к нему |
| ЛКМ (флаг в руке, наведено на ресурс, есть выделение) | — | Мгновенная задача «добывать»; флаг рассеивается |
| Встряхнуть рукой (с флагом) | — | Рассеять флаг; все выделенные → `free` |
| 1 | — | Назначить задачу «добывать» всем гоблинам в `waiting`; флаг на поле исчезает с эффектом |
| ЛКМ на замке (рука свободна) | — | Войти в режим стрельбы |
| ЛКМ (режим стрельбы) | — | Выстрел ядром в прицел |
| Q | Й | Выход из режима стрельбы |
| Z | Я | Отдалить камеру (×0.9) |
| X | Ч | Приблизить камеру (×1.1) |
| C | С | Сбросить зум |
| R | К | Полный перезапуск карты |

### Рестарт карты (R / К)
Функция `restartMap()` выполняет полный сброс:
- Все предметы возвращаются на начальные позиции (idle, iz=0)
- Миньоны возвращаются на начальные позиции (wandering)
- Флаг возвращается в состояние `docked`
- Рука сбрасывается (открыта, ничего не захвачено, `selectedMinions = []`)
- Камера сбрасывается (zoom = 1.0)
- Тряска экрана и эффект флага обнуляются
- Статус: «Карта перезапущена!»

## 9. Координатные системы

### Экранные (Screen)
- Пиксели от верхнего левого угла окна
- Используются для позиции мыши и рендеринга

### Изометрические (ISO)
- `ix`, `iy` — позиция на плоскости
- `iz` — высота над плоскостью (отдельная ось)

### Функции преобразования

| Функция | Вход | Выход | Описание |
|---------|------|-------|----------|
| `isoToScreen(ix, iy)` | ISO | Screen (canvas) | Базовое преобразование |
| `screenToIso(sx, sy)` | Screen (mouse) | ISO | Обратное, с учётом зума |
| `worldToScreen(wx, wy)` | ISO | Screen (centered) | isoToScreen + центрирование |
| `screenToCanvas(sx, sy)` | Screen (mouse) | Canvas (zoomed) | Для рендеринга в зумированном контексте |

## 10. Визуальные эффекты

- **Тени** — эллиптические, уменьшаются и бледнеют с высотой предмета
- **Подсветка наведения** — жёлтая рамка с пульсацией (`sin(time)`)
- **Покачивание** — предмет и рука покачиваются при переносе (sin/cos)
- **Тряска экрана** — при ударе тяжёлых предметов
- **Плавный подъём** — easeOut интерполяция при захвате
- **Рассеивание флага** — пиксели флага разлетаются с физикой и затухают за 0.5с (`flagEffect`)
- **Подсветка выделения** — зелёная пульсирующая рамка вокруг выделенных миньонов, пока флаг в руке
- **Тряска при уроне** — гоблин резко пошатывается при получении урона (`damageWobble` таймер, 0.4с, sin/cos шейк)
- **Частицы крови** — красные пиксели разлетаются из места удара (8 шт. при уроне, 20 при смерти), с гравитацией и затуханием по alpha
- **Лужица крови** — пиксельный паттерн на полу: при уроне (малая, 2с), при смерти (большая, 4с); рваные края с внешними брызгами; мёртвый гоблин крови не оставляет

## 11. UI

- Подсказка: "ЛКМ — схватить/бросить | Тяни на пустом месте — выделить миньонов" (фиксированная)
- Статус: динамический текст (Рука открыта / Навести / Захвачено / Флаг в руке / Флаг установлен / Флаг рассеян!)
- Индикатор зума (появляется при зуме ≠ 100%)
- Моноширинный шрифт, тень текста, без взаимодействия (`pointer-events: none`)

## 12. Коллизии между предметами

Функция `resolveItemCollisions()` вызывается каждый кадр после обновления физики всех предметов.

- Модель столкновения — упругие круги в изометрической плоскости
- Радиус каждого предмета задан в `ITEM_TYPES[].radius` (iso-единицы)
- Предметы в состоянии `carried` / `lifting` коллизии не участвуют
- Проверка по высоте: столкновение только если `|iz_a - iz_b| < (ra + rb) × 0.5`
- При коллизии: предметы раздвигаются пропорционально массам, обмениваются импульсом
- Коэффициент упругости = `min(bounciness_a, bounciness_b)`
- Неподвижный предмет (`idle` / `settling`) при ударе переходит в `sliding`
- Тряска экрана при сильных ударах (`impactSpeed > 2.0`)

## 13. Миньоны

2 миньона (зелёных гоблина, 6×8 px) на поле. Стейт-машина:

```
Геймплейные состояния (управление игроком):
  free ←────────────────────────────────────────────────────────┐
   │ лассо                                                       │ флаг рассеян / нет камней
   ↓                                                             │
  listening ──(флаг поставлен)──→ moving ──(достиг)──→ waiting ─┤
                                                         │ клавиша 1
                                                         ↓
                                                        busy ──(подобрал)──→ returning ──(сдал)──→ busy (или free)

Боевые состояния (гоблины автоматически):
  free / busy / returning → war ──(в радиусе удара)──→ fighting ──(победа)──→ <savedState>

Разведчик (goblinClass = 'scout'):
  free ←────────────────────────────── free (блуждает по всей карте, стареет 30с)
                                          │ scoutAge >= SCOUT_LIFESPAN
                                          ↓ (natural death)
                                         dead → skeleton (стандартный цикл)

Физические состояния (рука игрока):
  любое → lifting → carried → thrown → bouncing → sliding → settling → free (или dead)

Смерть и воскрешение:
  settling → dead ──(3 сек)──→ skeleton ──(бросок+приземление)──→ crumbled → (удаление)
```

### Геймплейные состояния

**Зоны патруля** (iso-тайлы от замка `(0, 0)`):
| Режим | Зона | Радиус | Поведение |
|-------|------|--------|-----------|
| `free` | 21×21 тайл | ±10 | `pickNewTarget()` выбирает случайную точку в этой зоне |
| `gathererMode` | 42×42 тайл | ±21 | `findNearestResource(items, zoneOnly=true)` ищет ресурсы только в этой зоне |

`gathererMode` определяется **географически** (по позиции ресурса в момент подбора):
- Ресурс в зоне ±21 тайла → `gathererMode = true` → после сдачи продолжает сбор в 42×42
- Ресурс за пределами зоны → `gathererMode = false` → после сдачи становится свободным (21×21)

| # | Состояние | Поведение |
|---|-----------|-----------|
| 1 | `free` | Непрерывно движется к случайной точке в зоне **21×21** вокруг замка; при достижении сразу выбирает новую. Базовое состояние |
| 2 | `listening` | Мгновенно останавливается. Активно когда выбран лассо и рука держит флаг |
| 3 | `moving` | Движется к `targetX/Y` (позиция флага). `targetX/Y` устанавливается при размещении флага |
| 4 | `waiting` | Стоит у флага, ждёт задачу от игрока. Переход из `moving` при достижении цели |
| 5 | `busy` | Идёт к ближайшему ресурсу. Переход из `waiting` по клавише **1**. Реагирует на скелетов в радиусе `GOBLIN_AGGRO_RANGE`. Если цель стала недоступна — ищет следующую или уходит в `free` |
| 6 | `returning` | Несёт ресурс к замку (`carriedItem.state = 'goblin_carried'`). Реагирует на скелетов в радиусе `GOBLIN_AGGRO_RANGE`. При достижении замка — сдаёт ресурс (удаляется), ищет следующий → `busy` или `free` |
| 7 | `war` | Видит скелета в радиусе `GOBLIN_AGGRO_RANGE = 10` (воин: `WARRIOR_AGGRO_RANGE = 25`) — движется к нему. Вход из `free`, `busy`, `returning`, `guarding`, `warrior_returning` через `enterCombat()` |
| 8 | `fighting` | В радиусе удара — атакует скелета. Базовый: 10 урона / 1.0с / дальность 0.6. Воин: 20 урона / 1.2с / дальность 0.8. При победе → `exitCombat()` |
| 9 | `guarding` | **Только воин.** Стоит на посту (`guardX/Y`), патрулирует радиус 0.3 тайла. Агрится на скелетов в радиусе 25 тайлов |
| 10 | `warrior_returning` | **Только воин.** Движется к позиции охраны `guardX/Y`. Переход в `guarding` при дистанции < 0.3 |

### Физические состояния (от GameObject)

| Состояние | Поведение |
|-----------|-----------|
| `lifting` / `carried` / `thrown` / `bouncing` / `sliding` | Та же физика, что у предметов (`GameObject.updatePhysics`) |
| `settling` | Пауза 0.3с, затем возврат в `free` или переход в `dead` если `hp <= 0` |
| `dead` | Не двигается; `deadTime` растёт — анимация Труп→Надгробие→Скелет; при `deadTime >= 3.0` → `skeleton` + `isUndead = true`. Используется для угасания тумана войны |
| `skeleton` | Скелет (`isUndead = true`): **враг** — бродит по всей карте, ищет гоблинов в радиусе 15 тайлов, атакует в ближнем бою (15 урона / 1.5с). Скорость 60% от гоблина. Не подчиняется игроку, не выделяется лассо, не раскрывает туман |
| `crumbled` | Конечное состояние скелета после броска; ожидает удаления из `minions[]` |

### Выбор лассо

Рамка выделяет гоблинов в состояниях `free`, `listening`, `moving`, `waiting`, `busy`, `returning`, `war`, `fighting`, `guarding`, `warrior_returning` (любые не-физические и не-`dead`). Выбранные → `listening`, рука берёт флаг.

Скелеты (`isUndead = true`) лассо **не выделяются** и не получают задач.

Воины (`goblinClass === 'warrior'`) **выделяются лассо** наравне с обычными гоблинами. При выделении переходят в `listening` и могут получить команду перемещения (см. раздел 13.5).

### Система здоровья гоблинов

| Константа | Значение | Описание |
|-----------|----------|----------|
| `MINION_MAX_HP` | 100 | Начальное HP |
| `FALL_DMG_MED_VZ` | 2.5 | Порог vz → средний урон |
| `FALL_DMG_HI_VZ` | 5.0 | Порог vz → большой урон |
| `FALL_DMG_MED` | 25 | Урон при среднем падении (25% HP) |
| `FALL_DMG_HI` | 50 | Урон при сильном падении (50% HP) |

Урон начисляется в хуке `onLand(impactVz)` при приземлении (кроме случая `dead === true`). При `hp <= 0` флаг `dead` устанавливается и после settling гоблин переходит в состояние `dead`.

**Скелет:** HP = 20 (`SKELETON_MAX_HP`). `onLand()` наносит урон от падения аналогично гоблину (25/50). При `hp <= 0` — `pendingBoneEffect`, `pendingRemove = true`, `state = 'crumbled'`.

**Параметры физики:** масса 0.7, отскок 0.25, трение 0.82

**Начальные позиции** (`INITIAL_MINION_POSITIONS`):
```
minions[0] @ (-4, 0)
minions[1] @ ( 4, 0)
```

**Взаимодействие с рукой:** захват и бросок аналогичны предметам — `hoveredMinion`, `hand.grabbedMinion`. При захвате или переназначении лассо вызывается `dropCarriedItem()` — гоблин бросает камень.

## 13.2. Система скелетов

Когда гоблин погибает (`hp <= 0`), он проходит через три визуальных фазы за 3 секунды, затем воскресает автономным скелетом.

### Анимация смерти (state = `dead`, таймер `deadTime`)

| Диапазон `deadTime` | Визуал | Переход |
|---------------------|--------|---------|
| 0 – 1.0 с | Труп (`MINION_DEAD_PIXELS`) | — |
| 1.0 – 1.4 с | Кросс-фейд труп → надгробие | `α = (t−1.0)/0.4` |
| 1.4 – 2.0 с | Надгробие (`TOMBSTONE_PIXELS`) | — |
| 2.0 – 2.4 с | Кросс-фейд надгробие → скелет | `α = (t−2.0)/0.4` |
| 2.4 – 3.0 с | Скелет (preview) | — |
| ≥ 3.0 с | → `state = 'skeleton'`, `isUndead = true` | автономный режим |

### Скелет как юнит (state = `skeleton`)

- Флаг `isUndead = true` — отличает скелета от гоблина во всех проверках
- **Враг** — ищет и атакует гоблинов (см. «Вражеское поведение» ниже)
- Блуждает по **всей карте** (`gameMap.size`), а не в зоне замка
- Скорость: `MINION_SPEED × 0.6` (`SKELETON_SPEED_FACTOR`)
- HP: `SKELETON_MAX_HP = 20` (получает урон от падения как гоблин)
- **Не** раскрывает туман войны (исключён из `fogSources`)
- **Не** выделяется лассо (`input.js` пропускает `isUndead`)
- **Не** является `hoveredMinion` в состояниях `dead` / `crumbled`
- **Можно** подобрать рукой и бросить — физика идентична гоблину

### Вражеское поведение скелетов

Скелеты — автономные враги, которые преследуют и атакуют гоблинов.

**Алгоритм (каждый кадр в стейте `skeleton`):**
1. Поиск ближайшего живого гоблина (не скелета, не мёртвого, не в руке) в радиусе `SKELETON_AGGRO_RANGE = 15` iso-тайлов
2. Если цель найдена и в радиусе удара (`SKELETON_ATTACK_RANGE = 0.6`) → атака:
   - Урон `SKELETON_ATTACK_DAMAGE = 15` HP
   - Кулдаун `SKELETON_ATTACK_CD = 1.5` секунды
   - Эффект тряски (`damageWobble`) + частицы крови (`pendingBloodEffect`)
   - При `hp <= 0` гоблин умирает → через 3 сек восстаёт скелетом (штатный цикл)
3. Если цель найдена, но далеко → движение к ней (скорость 60% гоблина)
4. Если целей нет → блуждание по карте (`pickNewTarget()` по всей карте)

**Лавинный эффект:** каждый убитый скелетом гоблин через 3 секунды становится новым скелетом, который тоже начинает охотиться — армия скелетов растёт экспоненциально.

| Константа | Значение | Описание |
|-----------|----------|----------|
| `SKELETON_MAX_HP` | 20 | HP скелета (разрушается при 0) |
| `SKELETON_AGGRO_RANGE` | 15.0 | Дистанция обнаружения гоблина (iso-тайлы) |
| `SKELETON_ATTACK_RANGE` | 0.6 | Дистанция удара (ближний бой) |
| `SKELETON_ATTACK_DAMAGE` | 15 | Урон за удар |
| `SKELETON_ATTACK_CD` | 1.5 | Кулдаун между ударами (секунды) |

### Разрушение скелета (`hp <= 0`)

При урон от падения / артиллерии / будущего боя гоблинов:
1. `pendingBoneEffect = { ix, iy }` — маркер для эффекта
2. `pendingRemove = true` — отложенное удаление
3. `state = 'crumbled'` — финальное состояние, рендер пропускается

В `main.js` за кадр: обходим `minions` по `pendingBoneEffect` → спауним 18 частиц в `bloodParticles` с `color` костей (`#d8d0bc` / `#c8c0ac` / `#e0d8c4` / `#b8b0a0`). Затем — обратный проход по `pendingRemove` со сдвигом `hand.grabbedMinion`.

### Спрайты

| Спрайт | Константы | Размер |
|--------|-----------|--------|
| `TOMBSTONE_PIXELS` | `TOMBSTONE_W=6`, `TOMBSTONE_H=9` | 6×9 px — серо-голубой камень с крестом |
| `SKELETON_PIXELS` | `SKELETON_W=6`, `SKELETON_H=8` | 6×8 px — кости с тёмными глазницами |

## 13.1. Система задач гоблинов

Гоблин в состоянии `waiting` может получить задачу от игрока. Задача хранится в поле `task` (null | `'gather'`).

### Задача «Добывать» (`gather`)

**Запуск (три способа):**
1. **Клавиша 1** — назначает задачу всем гоблинам в `waiting`. Флаг на поле исчезает с эффектом частиц (`triggerFlagEffectAtWorld`).
2. **Флаг в руке + наведение на ресурс + лассо-выделение → ЛКМ** — мгновенная задача без установки флага; флаг рассеивается у руки (`triggerFlagEffectAtHand`).
3. Ранее (устарело): установить флаг → гоблины ждут у флага → клавиша **1**.

**Поток выполнения:**
1. `assignGatherTask(items)` — находит ближайший доступный ресурс через `findNearestResource()` (фильтр по `typeDef.gatherable`), гоблин → `busy`
2. `busy` — гоблин идёт к ресурсу (`targetItem`). Если цель стала недоступна — ищет следующий. При приближении (`dist < 0.6`) и состоянии `idle`/`settling`/`sliding` — подбирает: предмет → `goblin_carried`, гоблин → `returning`
3. `returning` — несёт предмет к замку, предмет следует за гоблином (`iz = 0.5`). При достижении замка (`dist < baseRadius + radius`) — предмет удаляется из `items` через `splice`. Затем: если есть ещё ресурсы → `busy`, иначе → `free`

**Прерывание:** рука подбирает гоблина или лассо переназначает → `dropCarriedItem()` → `task = null`

### Состояние предмета `goblin_carried`

Новое состояние для предметов, которые несёт гоблин:
- Позиция управляется гоблином (обновляется в `returning`)
- Не участвует в коллизиях между предметами (`resolveItemCollisions`)
- Не участвует в коллизии с замком (`Castle.pushObjects`)
- Не доступен для захвата рукой (hover detection в `main.js`)
- Физика (`Item.update`) пропускается

### Поля `Minion` для задачи

| Поле | Тип | Описание |
|------|-----|----------|
| `task` | null \| `'gather'` | Текущая задача |
| `targetItem` | Item \| null | Ссылка на предмет-цель (не индекс!) |
| `carriedItem` | Item \| null | Ссылка на переносимый предмет |
| `gathererMode` | boolean | `true` — ресурс был в зоне ±21, после сдачи ищет следующий в 42×42; `false` — после сдачи уходит в `free` |
| `deadTime` | number | Секунды с момента смерти; используется для плавного угасания зоны видимости в тумане войны |

Объектные ссылки (не индексы) защищают от смещения индексов при `items.splice()`.

## 13.3. Боевая система гоблинов

Гоблины автоматически атакуют скелетов, не требуя команды игрока.

### Проактивный агро (`_tryAggro`)

Метод `_tryAggro(allMinions)` вызывается каждый кадр из состояний `free`, `busy` и `returning`.

**Алгоритм:**
1. Перебирает всех миньонов, пропуская не-скелетов (`!m.isUndead`) и не-активных (`state !== 'skeleton'`)
2. Находит ближайшего скелета в радиусе `GOBLIN_AGGRO_RANGE = 10` iso-тайлов
3. Если нашёл — вызывает `enterCombat(target)` и возвращает `true`

### Вход в бой (`enterCombat`)

При обнаружении врага гоблин:
1. Сохраняет текущий контекст: `savedState`, `savedTask`, `savedTargetItem`, `savedGathererMode`
2. Если нёс ресурс — `dropCarriedItem()` (ресурс падает на землю)
3. Устанавливает `combatTarget = target`, переходит в `state = 'war'`

### Преследование (`war`) и атака (`fighting`)

- `war`: движется к `combatTarget` со скоростью `MINION_SPEED`. При сближении (`dist < GOBLIN_ATTACK_RANGE`) → `fighting`
- `fighting`: атакует каждый `GOBLIN_ATTACK_CD` сек, наносит `GOBLIN_ATTACK_DAMAGE` урона. Отступает в `war` если цель вышла из радиуса удара

**Валидность цели** (`_isValidTarget`): цель жива, не `pendingRemove`, не `crumbled`

### Выход из боя (`exitCombat`)

При победе (цель уничтожена) или потере цели:
1. `combatTarget = null`, `attackCooldown = 0`
2. Восстанавливает состояние по `savedState`:
   - `'gather'` → возобновляет сбор (`busy`, ищет новый ресурс)
   - `'listening'` / `'waiting'` / `'moving'` → возвращается к назначенному игроком состоянию
   - иначе → `free`, `pickNewTarget()`

### Rally-система (призыв на помощь)

Когда гоблин получает урон от скелета, он вызывает союзников:
- Все свободные гоблины (`state = 'free'`) в радиусе `GOBLIN_RALLY_RANGE = 15` iso-тайлов
- Переходят в `war` с тем же `combatTarget`

### Константы боевой системы гоблинов

| Константа | Значение | Описание |
|-----------|----------|----------|
| `GOBLIN_ATTACK_DAMAGE` | 10 | Урон за удар |
| `GOBLIN_ATTACK_CD` | 1.0 с | Кулдаун между ударами |
| `GOBLIN_ATTACK_RANGE` | 0.6 | Дистанция удара (iso-тайлы) |
| `GOBLIN_AGGRO_RANGE` | 10 | Дистанция обнаружения скелетов (iso-тайлы) |
| `GOBLIN_RALLY_RANGE` | 15 | Дистанция призыва союзников при получении урона |

### Конфликт ресурсов (`findNearestResource`)

`findNearestResource(items, zoneOnly, allMinions)` теперь принимает список всех миньонов. При `allMinions != null` пропускает ресурс, к которому уже движется другой гоблин (`m.state === 'busy' && m.targetItem === item`). Предотвращает «столпотворение» — несколько гоблинов идут к одному предмету.

## 13.4. Класс воина (`goblinClass = 'warrior'`)

### HUD производства воинов (`main.js`, screen space)

Кликабельный блок под HUD производства гоблинов. Структура аналогична гоблинскому блоку:
- Иконка воина: спрайт `MINION_PIXELS` (scale=2) + шлем `WARRIOR_HELMET_PIXELS` поверх
- Счётчик `воины_в_строю / всего_живых_гоблинов`
- Прогресс-бар золотого цвета (`#cc9922`) — показывает прогресс таймера `warriorUpgradeTimer` при активном производстве
- Кнопка `▶ Запустить` / `⏸ Остановить`

Клик по блоку → `warriorProduction.active = !warriorProduction.active` (перехватывается в capture phase).

`tryUpgradeWarrior()` выполняется только если `warriorProduction.active === true`.

### Апгрейд

Каждые `WARRIOR_UPGRADE_INTERVAL = 8` секунд в `main.js` вызывается `tryUpgradeWarrior()`:
1. Проверяет: `castleResources[3] >= 1` (есть железо)
2. Собирает всех свободных базовых гоблинов (`state = 'free'`, `goblinClass = 'basic'`, не мёртв)
3. Выбирает случайного, снимает 1 железо: `castleResources[3]--`
4. Устанавливает `goblinClass = 'warrior'`, получает позицию охраны через `getNextWarriorGuardPos()`
5. Гоблин идёт к посту: `state = 'warrior_returning'`

### Стена воинов (`World.js`)

`getNextWarriorGuardPos()` возвращает следующую позицию охраны вдоль окружности радиуса `WARRIOR_GUARD_RADIUS = 50` тайлов от замка:
```
angle = warriorWall.baseAngle + warriorWall.count * WARRIOR_WALL_STEP
ix = cos(angle) * 50,  iy = sin(angle) * 50
```
- `warriorWall.baseAngle` — случайный угол при старте; сбрасывается при рестарте (`resetWarriorWall()`)
- `WARRIOR_WALL_STEP = 0.04` рад ≈ 2 тайла между соседними воинами на периметре

Воины выстраиваются в линию, образуя «живую стену» вдоль одного направления от замка.

### Поведение воина

| Состояние | Поведение |
|-----------|-----------|
| `warrior_returning` | Движется к `guardX/Y` со скоростью `MINION_SPEED`. Агрится при обнаружении скелетов. При дистанции < 0.3 → `guarding` |
| `guarding` | Стоит на посту. Каждый кадр `_tryAggro()` в радиусе 25 тайлов. При сдвиге > 0.3 — корректирует позицию |

**После боя:** `exitCombat()` для воина всегда → `warrior_returning` (возврат на пост), игнорируя `savedState`.

**После броска рукой:** `onSettle()` → `warrior_returning` (воин не подбирает ресурсы).

### Rally воинов

При нападении скелета на любого гоблина (`skeleton` fighting state):
- Обычные гоблины в `GOBLIN_RALLY_RANGE = 15` тайлов → `enterCombat`
- Воины в `WARRIOR_GUARD_RADIUS = 50` тайлов → `enterCombat` (бегут с любой точки периметра)

### Визуал

Шлем (`WARRIOR_HELMET_PIXELS`, 6×3 px) отрисовывается поверх головы гоблина в `draw()` если `goblinClass === 'warrior' && !isUndead`. Металлические тона: `#7788aa` / `#aabbcc` / `#bbccdd` с бликом `#ccddee`.

Воины **выделяются лассо** и получают команду перемещения в кольцевой строй (см. раздел 13.5).

| Константа | Значение | Описание |
|-----------|----------|----------|

| `WARRIOR_AGGRO_RANGE` | 25 | Дистанция обнаружения врагов |
| `WARRIOR_ATTACK_DAMAGE` | 20 | Урон за удар (в 2× сильнее базового) |
| `WARRIOR_ATTACK_CD` | 1.2 с | Кулдаун между ударами |

| `WARRIOR_IRON_COST` | 1 | Железо на апгрейд |

## 13.6. Класс разведчика (`goblinClass = 'scout'`)

Гоблин-разведчик — временный юнит с расширенной зоной видимости. Создаётся трансформацией свободного базового гоблина, живёт 30 секунд и умирает естественной смертью → превращается в скелет по стандартному циклу.

### HUD производства разведчиков (`main.js`, screen space)

Кликабельный блок под HUD производства воинов. Зелёная рамка (`#33aa44`). Структура:
- Иконка: спрайт `MINION_PIXELS` (scale=2) + капюшон `SCOUT_HOOD_PIXELS` поверх
- Счётчик `разведчиков_живых / SCOUT_MAX_COUNT` (макс 5)
- Прогресс-бар зелёного цвета (`#33aa44`) — показывает таймер `scoutUpgradeTimer` при активном производстве
- Стоимость: 2 дерева + 1 пшеница + базовый гоблин

Клик по блоку → `scoutProduction.active = !scoutProduction.active`.

### Цикл производства

`tryUpgradeScout()` вызывается каждые `SCOUT_UPGRADE_INTERVAL = 10` секунд:
1. Проверяет: `scoutProduction.active`, количество живых разведчиков < `SCOUT_MAX_COUNT = 5`
2. Проверяет ресурсы: `castleResources[2] >= 4` (дерево) и `castleResources[0] >= 1` (пшеница)
3. Собирает свободных базовых гоблинов (`state = 'free'`, `goblinClass = 'basic'`, не мёртв)
4. Выбирает случайного, снимает 2 дерева и 1 пшеницу
5. Устанавливает `goblinClass = 'scout'`, `scoutAge = 0` — гоблин немедленно начинает блуждать

### Поведение

| Свойство | Значение |
|----------|----------|
| Зона блуждания | Вся карта (`gameMap.size`, как скелеты) — не ограничен 21×21 |
| Зона видимости | `SCOUT_FOG_RADIUS = 5` iso-тайлов (2.5× normal) |
| Жизнь | `SCOUT_LIFESPAN = 30.0` секунд |
| Старение | `scoutAge += dt` только в активных состояниях — **не** в физических (`lifting`, `carried`, `thrown`, `bouncing`, `sliding`) |
| Сбор ресурсов | **Запрещён** — `onSettle()` и `input.js` пропускают разведчика |
| Боевые задачи | Нет — разведчик не агрится на скелетов (нет вызова `_tryAggro`) |
| После смерти | Стандартный `dead → skeleton` цикл; скелет наследует `goblinClass = 'scout'`, но `isUndead = true` исключает его из всей логики разведчика |

### Смерть по старости

```js
// В update(), только активные состояния (не физика):
if (this.goblinClass === 'scout' && !this.dead && !this.isUndead && !physicsStates.includes(this.state)) {
    this.scoutAge += dt;
    if (this.scoutAge >= SCOUT_LIFESPAN) {
        this.dropCarriedItem(); // освобождает ресурс если нёс
        this.dead = true;
        this.state = 'dead';
        this.stateTime = 0;
        this.deadTime = 0;
        return;
    }
}
```

### Исключения из стандартной логики

| Место | Что исключено |
|-------|---------------|
| `Minion.onSettle()` | После приземления не берёт ресурс — `pickNewTarget()`, `state = 'free'` |
| `input.js` — флаг+ресурс | При назначении задачи «добывать» разведчик пропускается |
| `input.js` — флаг+место | При установке флага разведчик пропускается (не переходит в `moving`) |
| `Castle.js` aliveCount | Скелет-бывший-разведчик исключён через `isUndead` |
| Fog sources | Радиус 5 (не 2) если `goblinClass === 'scout'` |

### Визуал

Капюшон `SCOUT_HOOD_PIXELS` (6×3 px, тёмно-зелёный `#3a6030`/`#233d1c`) отрисовывается поверх головы гоблина если `goblinClass === 'scout' && !isUndead`. В панели выделенных юнитов (selection bar) — гоблин с капюшоном.

### Константы

| Константа | Значение | Описание |
|-----------|----------|----------|
| `SCOUT_MAX_COUNT` | 5 | Макс. одновременно живых разведчиков |
| `SCOUT_LIFESPAN` | 30.0 | Секунд до естественной смерти |
| `SCOUT_FOG_RADIUS` | 5.0 | Радиус зрения в iso-тайлах |
| `SCOUT_UPGRADE_INTERVAL` | 10.0 | Секунд между производственными циклами |
| `SCOUT_WOOD_COST` | 2 | Единиц дерева (typeIndex 2) |
| `SCOUT_FOOD_COST` | 1 | Единиц пшеницы (typeIndex 0) |

## 13.5. Ручное управление воинами

Воины выделяются лассо так же, как обычные гоблины, но получают особую команду — перемещение в **кольцевой строй** вокруг целевой точки.

### Выделение

- Рамкой лассо захватываются воины в состояниях `guarding` и `warrior_returning` (добавлены в `SELECTABLE`)
- При выделении воин переходит в `listening` (останавливается), рука берёт флаг
- Выделение воинов и обычных гоблинов может быть смешанным

### Команда перемещения (флаг в руке → ЛКМ на свободном месте)

Воины и обычные гоблины обрабатываются раздельно:

| Тип | Поведение |
|-----|-----------|
| Обычный гоблин | `state = 'moving'`, `targetX/Y = flag.ix/iy` — идут прямо к флагу |
| Воин | Получает уникальную позицию в кольце; `guardX/Y` обновляется, `state = 'warrior_returning'` |

**Кольцевой строй (воины):**
- N воинов равномерно распределяются по окружности вокруг точки флага
- `radius = max(5, N × 10 / 2π)` — шаг дуги между соседними ≈ 10 тайлов
- `angle[i] = 2π × i / N`; позиция: `(flag.ix + radius × cos(angle), flag.iy + radius × sin(angle))`
- При N = 1 воин идёт прямо в точку (radius = 0)
- После прихода воин переходит в `guarding` на новом посту (стандартный механизм `warrior_returning`)

### Команда добычи (флаг в руке + наведено на ресурс)

Воины **не добывают ресурсы**. При выдаче задачи «добывать» через клик на ресурс:
- Обычные гоблины → `assignGatherTask()`
- Воины → сразу `state = 'warrior_returning'` (возвращаются на свой текущий пост охраны)

## 14. Флаг и управление миньонами

Объект `flag` с состояниями: `docked` | `carried` | `placed`.

### Флаг (pixel-art, 5×8 px)
- Древко + жёлтое полотно (`FLAG_PIXELS`)
- Рисуется у курсора с покачиванием, пока `hand.grabbedFlag === true`

### Поток взаимодействия
1. Игрок тянет рамку выделения на пустом поле (от позиции руки)
2. Если попало ≥ 1 миньона — они переходят в `listening`, рука берёт флаг (`hand.grabbedFlag = true`), выделенные подсвечиваются зелёной рамкой
3. **ЛКМ на свободном месте** — флаг устанавливается (`flag.state = 'placed'`), `listening` → `moving` с `targetX/Y = flag.ix/iy`
4. Гоблин достигает флага → `moving` → `waiting` (стоит у флага, ждёт задачу)
5. **Встряхнуть рукой** — флаг рассеивается с эффектом частиц; `listening`/`moving`/`waiting` → `free`

### Подбор поставленного флага

**ЛКМ на поставленном флаге** (рука свободна, `hoveredFlag = true`):
- `flag.state = 'placed'` → `'docked'`; `hand.grabbedFlag = true`
- Все гоблины в `moving`/`waiting` → `listening`; добавляются в `hand.selectedMinions`
- Флаг можно переставить в новое место — гоблины пойдут туда

**Hover-детектирование флага** (`hoveredFlag`):
- Активно только когда рука свободна и `flag.state === 'placed'`
- Радиус: 1.2 iso-единицы от `flag.ix/iy`
- Флаг подсвечивается жёлтой рамкой с пульсацией (уже существующий `drawFlagAt(isHovered)`)
- Приоритет: только если `hoveredItem === null && hoveredMinion === null`

### Мгновенная задача «добывать» (флаг в руке + ресурс)

Когда `hand.grabbedFlag === true`, в `update()` активируется hover-детектирование добываемых ресурсов:
- Ищет ближайший предмет с `typeDef.gatherable === true` в радиусе 1.5 iso-единиц
- При наведении: ресурс подсвечивается жёлтой рамкой, статус — подсказка о клике

**ЛКМ** при `hoveredItem !== null && selectedMinions.length > 0`:
- Все выбранные гоблины в `listening` → `assignGatherTask(items)`
- Флаг рассеивается эффектом частиц у руки (`triggerFlagEffectAtHand`)
- `hand.grabbedFlag = false`, `selectedMinions = []`

### Панель выделенных юнитов (Selection Bar)

Пока `hand.grabbedFlag === true` и `hand.selectedMinions.length > 0` — в нижнем центре экрана отображается панель с иконками выбранных юнитов.

**Отрисовка (screen-space, после ctx.restore()):**
- Слоты шириной `MINION_W × 2 + 12` px, высотой `MINION_H × 2 + 12` px с зазором 4 px
- Панель центрируется горизонтально, отступ 32 px от нижнего края
- Каждый слот: спрайт гоблина/воина в масштабе ×2 (воин — с шлемом)
- **На hover:** фон `#552200`, рамка 2 px `#ff6600`, спрайт притушен (alpha 0.35) + символ `✕`

**Клик по иконке (capture phase, до input.js):**
- Обычный гоблин: `pickNewTarget()`, `state = 'free'`
- Воин: `state = 'warrior_returning'` (идёт на свой пост охраны)
- Юнит убирается из `hand.selectedMinions`
- Если `selectedMinions` опустел — флаг убирается (`flag.state = 'docked'`, `hand.grabbedFlag = false`)

**Блокировка edge scroll:**
- Rect панели (`selectionBarPanel`) обновляется каждый кадр при рендере
- `selectionBarPanel.w = 0` когда панель скрыта — отключает блокировку
- В `update()` в условии `overHud` панель учитывается наряду с правым HUD-панелью

### Детектор встряхивания
- Каждый кадр записывается дельта экранного X руки в `hand.shakeHistory` (кольцевой буфер за 0.6с)
- Подсчитываются смены знака — порог: **5 смен за 0.6с**
- Срабатывает `dismissFlag()`

### Эффект рассеивания флага (`flagEffect`)
- Каждый пиксель `FLAG_PIXELS` превращается в частицу со случайной скоростью
- Гравитация частиц: +120 ед/с²
- Затухание по альфа за `duration = 0.5с` (степенная кривая)
- Координаты хранятся в canvas pre-zoom space (через `screenToCanvas`)

Три точки запуска:
| Функция | Когда вызывается | Источник координат |
|---------|-----------------|-------------------|
| `triggerFlagEffectAtHand()` | Встряхивание, задача через флаг+ресурс | `screenToCanvas(hand.screenX, hand.screenY)` |
| `triggerFlagEffectAtWorld(ix, iy)` | Клавиша 1 с флагом на поле | `worldToScreen(ix, iy)` |
| — (внутри `dismissFlag`) | Любое рассеивание флага в руке | вызывает `triggerFlagEffectAtHand()` |

## 15. Замок

Статичный объект на поле, расположен в iso-позиции `(0, 0)` — центр карты.

### Спрайт
- 10×12 px, масштаб 8 → 80×96 px (ровно 4× высота гоблина 24px)
- Башня с зубцами, ворота, пушка в бойнице
- Пушка (`CASTLE_CANNON_PIXELS`, 4×2 px) рисуется отдельно поверх спрайта с анимацией отдачи

### Рендеринг
- Один entry в renderList с `depth = ix + iy`
- Тень — эллипс под основанием замка
- Рука всегда поверх замка (`depth: Infinity`)

### Коллизия
- Круговая в iso-плоскости, радиус `CASTLE_BASE_RADIUS = 1.3` iso-единиц
- Высота башни `CASTLE_TOWER_HEIGHT = 4.0` — объекты выше не сталкиваются
- Предметы и миньоны в состоянии `carried`/`lifting`/`goblin_carried` коллизию не проходят
- Миньоны в состоянии `returning` коллизию не проходят — чтобы войти в зону замка и сдать камень
- При столкновении объект выталкивается наружу + отражение скорости по нормали (`WALL_BOUNCE`)

| Константа | Значение | Описание |
|-----------|----------|----------|
| `CASTLE_BASE_RADIUS` | 1.3 | Радиус коллизии (iso-единицы) |
| `CASTLE_TOWER_HEIGHT` | 4.0 | Максимальная высота коллизии (world units) |

### Производство гоблинов

Замок производит гоблинов автоматически. Управление через HUD (кликабельный блок под счётчиком ресурсов).

| Константа | Значение | Описание |
|-----------|----------|----------|
| `GOBLIN_MAX` | 25 | Максимум живых гоблинов |
| `GOBLIN_SPAWN_DURATION` | 3.0 с | Время производства одного гоблина |
| `GOBLIN_FOOD_COST` | 1 | Единиц пшеницы за гоблина |
| `GOBLIN_FOOD_TYPE` | 0 | Индекс пшеницы в `ITEM_TYPES` |

**Логика производства (`Castle.update`):**
- **Старт:** если `production.active`, `aliveCount < 25` и есть пшеница → списывается 1 еды, начинается 3-секундный таймер
- **Прогресс:** уже начатый таймер продолжается **независимо** от того, заполнен замок или нет
- **Завершение:** по окончании таймера — если `aliveCount < 25` → `pendingSpawn = true` (гоблин заспавнится); если замок полон — гоблин готов, но не появляется. Еда потрачена в момент старта, не возвращается
- По завершении `pendingSpawn`: `main.js` вызывает `spawnMinion(castle.spawnPoint)` → новый Minion добавляется в `minions[]`
- **Пауза:** если игрок нажимает кнопку при `progress > 0` — текущий гоблин **доделывается** (не прерывается), следующий не начинается

**HUD производства (screen space, под ресурсами):**
- Иконка гоблина (спрайт `MINION_PIXELS`, scale=2)
- Счётчик `живых / 25`
- Прогресс-бар (зелёный = производство идёт; оранжевый = завершает текущего на паузе)
- Статус: `▶ Запустить` / `⏸ Остановить` (лейбл показывает действие по клику, не текущее состояние)
- Клик по блоку → `production.active = !production.active` (перехватывается в capture phase до `input.js`)
- Рамка подсвечивается при наведении (`#aaaaff`), фон светлеет, курсор меняется на `pointer`
- Весь HUD-панель (ресурсы + производство) блокирует edge scroll камеры — `hudPanelRect` проверяется в `update()` перед применением панорамирования
- Отступ от краёв экрана: `HUD_MARGIN = 40px` (≈1 см)

**Спавн:** `World.spawnMinion(ix, iy)` — единственная точка создания новых миньонов вне `initWorld`/`restartMap`. Гоблин появляется у `castle.spawnPoint` (ворота замка: `iy + baseRadius + 0.5`).

### Архитектура (масштабируемость)
- `vx / vy` — готовы для будущего движения замка
- `fireCannon()` / `cannon.recoilTimer` — готовы для будущей системы атак
- `spawnPoint` getter — точка появления миньонов из ворот (`iy + baseRadius + 0.5`)
- `castle` экспортируется как `export let` из `World.js` — переприсваивается при рестарте

## 15.1. Артиллерия замка

Замок может стрелять ядрами по карте. Игрок входит в режим стрельбы, наводит изометрический прицел и делает выстрел по артиллерийской дуге.

### Вход в режим стрельбы

- Рука свободна + ЛКМ на замке (расстояние < `ARTILLERY_GRAB_RADIUS = 2.0` iso)
- Рука фиксируется на замке (`hand.state = 'closed'`), iso-координаты = замок
- Появляется изометрический прицел, следующий за мышью
- Камера управляется edge scroll (мышь у края экрана), зум — клавишами Z/X/C

### Стейт-машина артиллерии (`artilleryMode.state`)

```
aiming → (ЛКМ) → flying → (приземление) → aftermath → (1.5с) → aiming
                                                        ↑
  (Q / Й) → выход из режима в любом состоянии ──────────┘
```

| Состояние | Поведение |
|-----------|-----------|
| `aiming` | Прицел следует за мышью (`screenToIso`). Камера управляется edge scroll (мышь у края экрана). Зум — ручной (Z/X/C) |
| `flying` | Снаряд летит по параболе. Камера следует за ядром (lerp ×4). При `iz ≤ 0` → взрыв |
| `aftermath` | Частицы взрыва обновляются. Через 1.5с камера плавно возвращается к замку (lerp ×3), затем → `aiming` |

### Траектория снаряда

При выстреле рассчитывается время полёта `T = clamp(dist × 0.12, 0.6, 2.5)`:
- `vx = dx / T`, `vy = dy / T` — горизонтальная скорость (линейная)
- `vz = 0.5 × GRAVITY × T` — начальная вертикальная скорость (парабола приземляется точно в цель)
- В полёте: `vz -= GRAVITY × dt`, стандартная физика

### Камера в режиме артиллерии

Формула центрирования: `camera.x = isoToScreen(target).x × zoom`, `camera.y = isoToScreen(target).y × zoom`

| Фаза | Цель камеры | Зум |
|------|-------------|-----|
| `aiming` | Edge scroll (мышь у края экрана) | Ручной (Z/X/C), сохраняется между фазами |
| Момент выстрела | Остаётся на месте | Без изменений |
| `flying` | Снаряд (плавно, lerp ×4) | Без изменений |
| `aftermath` (0–1.5с) | Место взрыва (неподвижна) | Без изменений |
| `aftermath` (>1.5с) | Замок (плавно, lerp ×3) | Без изменений |

Рука в артиллерии: экранная позиция = `isoToScreen(castle) × zoom + canvasCenter - camera`.

### Взрыв

При приземлении ядра (`iz ≤ 0`, `timer > 0.05`):
1. **Тряска экрана** — интенсивность 15
2. **Урон** — все миньоны в радиусе `ARTILLERY_BLAST_RADIUS = 3` iso-тайлов получают 80 HP урона
3. **Отбрасывание** — сила `(1 - dist/radius) × 6`, по направлению от эпицентра; вертикальный импульс `pushForce × 0.5`
4. **Частицы** — 60 огненных + 30 земляных/обломков, гравитация 120, затухание за 0.5–1.1с
5. **Вспышка** — жёлтый эллипс, затухает за 0.15с
6. **Гарь** — тёмный эллипс на земле, появляется через 0.2с, затухает к концу `duration = 1.0`

## 16.1. Опыт разработки: найденные баги и архитектурные уроки

Этот раздел фиксирует реальные баги, найденные в процессе разработки, и архитектурные выводы из них. Служит справочником при расширении кодовой базы.

### Баг 1 — Скелеты блокировали производство гоблинов (критический)

**Файл:** `Castle.js` → `update()`

**Симптом:** После первой смерти гоблина (превращения в скелет) замок переставал производить новых гоблинов, даже если пшеница была.

**Причина:** Фильтр живых гоблинов не исключал скелетов (`isUndead = true`):
```js
// Было (сломано):
const aliveCount = minions.filter(m => m.state !== 'dead').length;
```
Скелеты имеют `state = 'skeleton'` (не `'dead'`), поэтому они засчитывались как живые и занимали слоты.

**Исправление:**
```js
// Стало:
const aliveCount = minions.filter(m => m.state !== 'dead' && !m.isUndead).length;
```

**Архитектурный урок:**
Фильтры «живых» объектов должны учитывать **все** флаги нежизнеспособности (`isUndead`, `pendingRemove`, etc.), а не только проверку `state`. При добавлении нового типа юнита (нежить, призраки) нужно явно решить — занимают ли они слоты в счётчиках. Этот инвариант стоит документировать в комментарии рядом с `aliveCount`.

### Баг 2 — Краш при доставке предмета в замок (критический)

**Файл:** `Minion.js` → `update()`, ветка `'returning'`

**Симптом:** Если игрок держал в руке предмет, а гоблин-сборщик одновременно доставлял другой предмет в замок — мог возникнуть краш или рука начинала управлять случайным предметом.

**Причина:** При `splice(idx, 1)` из массива `items[]` код корректировал `hand.grabbedItem` только для случая `> idx`, но не для случая `=== idx`:
```js
// Было (сломано):
if (hand.grabbedItem !== null && hand.grabbedItem > idx) {
    hand.grabbedItem--;
}
// Случай hand.grabbedItem === idx не обрабатывался → dangling index
```

**Исправление:**
```js
// Стало:
if (hand.grabbedItem !== null) {
    if (hand.grabbedItem === idx) {
        hand.grabbedItem = null;          // предмет исчез — рука пуста
    } else if (hand.grabbedItem > idx) {
        hand.grabbedItem--;               // сдвиг индекса после splice
    }
}
```

**Архитектурный урок:**
Паттерн «индекс в массиве» хрупок. При каждом `splice` нужно обрабатывать **три случая**: `< idx` (без изменений), `=== idx` (обнулить), `> idx` (декремент). Более надёжная альтернатива — хранить ссылку на объект, а не индекс: `hand.grabbedItemRef = item` вместо `hand.grabbedItem = 42`. Тогда `splice` не ломает ссылку, и не нужны никакие корректировки.

### Баг 3 — Некорректная скорость броска флага (логический)

**Файл:** `input.js` → обработчик `mouseup`, ветка выделения лассо

**Симптом:** После выделения гоблинов лассо флаг "запоминал" скорость предыдущего броска и мог улетать в сторону при следующем отпускании.

**Причина:** При передаче флага в руку через лассо `velocityHistory` не сбрасывался:
```js
// Было (сломано): velocityHistory не очищался
hand.grabbedFlag = true;
hand.state = 'closing';
hand.animProgress = 0;
// velocityHistory = [] — забыли
```

**Исправление:**
```js
hand.grabbedFlag = true;
hand.state = 'closing';
hand.animProgress = 0;
hand.velocityHistory = [];   // ← добавлено
```

**Архитектурный урок:**
Сброс `velocityHistory` — инвариант при любом «захвате» руки. Все три пути захвата (предмет, миньон, флаг) должны его очищать. Это хороший кандидат для метода `hand.grab(target)`, который централизует сброс состояния и исключает подобные пропуски.

### Баг 4 — Производительность: `flatMap` в игровом цикле (некритический)

**Файл:** `main.js` → `gameLoop()`, сборка `fogSources`

**Симптом:** Незначительное снижение FPS при большом числе миньонов из-за лишних аллокаций.

**Причина:** `flatMap` создавал новый массив на каждый кадр (60 раз/сек), что увеличивало нагрузку на GC:
```js
// Было:
const fogSources = [
    castleSource,
    ...minions.flatMap(m => { /* возвращает [] или [source] */ }),
];
```

**Исправление:** Заменено на императивный `for...of` с `push`:
```js
// Стало:
const fogSources = [castleSource];
for (const m of minions) {
    if (...) continue;
    fogSources.push({ ix: m.ix, iy: m.iy, radius: 2 });
}
```

**Архитектурный урок:**
В hot path игрового цикла (60+ вызовов/сек) избегать функций высшего порядка (`map`, `filter`, `flatMap`, `reduce`), которые аллоцируют промежуточные массивы. Допустимо использовать их в разовых операциях (инициализация, рестарт), но не в `update()`/`render()`. Профиль: вместо `flatMap` → `push` в loop.

### Баг 5 — Мёртвый код в `Hand` (некритический)

**Файл:** `Hand.js` → `constructor()`

**Симптом:** Нет симптома — просто мусор в коде.

**Причина:** Поля `this.targetX` и `this.targetY` объявлялись в конструкторе, устанавливались в нескольких местах, но нигде не читались. Вероятно, остаток от ранней идеи «рука следует к цели».

**Исправление:** Поля удалены из конструктора.

**Архитектурный урок:**
Писать-не-читать поля — признак незавершённой рефакторинга. При добавлении нового поля сразу фиксировать в комментарии, где оно читается. Периодически делать поиск по полю (`grep`) перед тем как оставить его в коде.

### Баг 6 — Stale-поля при переходе гоблина в скелета (логический)

**Файл:** `Minion.js` → `update()`, ветка `'dead'`

**Симптом:** Скелет после воскрешения мог немедленно войти в бой, вернуться к несуществующей задаче или двигаться к ресурсу, который уже подобран.

**Причина:** При переходе `dead → skeleton` не очищались поля от прошлой жизни гоблина: `combatTarget`, `savedState`, `task`, `targetItem`, `carriedItem`, `gathererMode`.

**Исправление:** После установки `state = 'skeleton'` принудительно обнуляются все эти поля и вызывается `pickNewTarget()` заново.

**Архитектурный урок:**
Каждый переход в «новую личность» (гоблин → скелет) должен явно объявлять, какие поля принадлежат старой личности. Инициализация через конструктор не помогает для юнитов, меняющих тип в рантайме — нужен явный reset-блок при каждом таком переходе.

### Баг 7 — Несколько гоблинов идут к одному ресурсу (логический)

**Файл:** `Minion.js` → `findNearestResource()`

**Симптом:** Несколько сборщиков одновременно двигались к одному предмету, толпились, потом трое возвращались без добычи (ресурс мог взять только один).

**Причина:** `findNearestResource` не проверяла, занят ли ресурс другим гоблином. Каждый гоблин видел одни и те же доступные ресурсы.

**Исправление:** Передаётся `allMinions` в метод. Ресурс помечается «занятым», если `m.state === 'busy' && m.targetItem === item` для любого другого гоблина.

**Архитектурный урок:**
Поиск ресурсов — задача с разделяемым состоянием. Резервирование цели через ссылку (`targetItem`) — простейший способ избежать коллизий без глобальной очереди задач.

### Баг 8 — exitCombat не восстанавливал назначенные игроком состояния (логический)

**Файл:** `Minion.js` → `exitCombat()`

**Симптом:** Гоблин, получивший задачу (флаг, `listening`/`waiting`/`moving`), после победы над скелетом уходил в `free` и терял назначение.

**Причина:** `exitCombat` обрабатывал только `savedTask === 'gather'` и дефолтный случай `free`. Состояния `listening`/`waiting`/`moving` не были предусмотрены.

**Исправление:** Добавлена явная ветка: если `savedState` — одно из управляемых игроком состояний, гоблин в него возвращается.

**Архитектурный урок:**
При сохранении состояния (`savedState`) нужно заранее определить **все** возможные значения и обработать каждый при восстановлении. Дефолт `free` должен быть явным fallback, не единственным путём.

### Баг 10 — scoutAge росло пока гоблин в руке (логический)

**Файл:** `Minion.js` → `update()`, счётчик возраста разведчика

**Симптом:** Разведчик мог умереть пока игрок держит его в руке — труп оказывался в руке.

**Причина:** Условие старения не проверяло физические состояния:
```js
// Было (сломано):
if (this.goblinClass === 'scout' && !this.dead && !this.isUndead) {
    this.scoutAge += dt; // росло даже в 'carried'
```

**Исправление:**
```js
// Стало:
if (this.goblinClass === 'scout' && !this.dead && !this.isUndead
    && !physicsStates.includes(this.state)) {
    this.scoutAge += dt;
```

**Архитектурный урок:**
Таймеры «жизни» юнита должны учитывать физические состояния — пока юнит в руке игрока, он «заморожен». Проверка `physicsStates.includes(state)` — готовый паттерн для любого подобного таймера.

### Баг 11 — Ресурс зависал при смерти разведчика по старости (логический)

**Файл:** `Minion.js` → `update()`, блок смерти по истечению `SCOUT_LIFESPAN`

**Симптом:** Если разведчик нёс ресурс в состоянии `returning` (нетипично, но возможно) и умер по старости — предмет оставался в состоянии `goblin_carried` навсегда: не падал, не был доступен.

**Причина:** При установке `dead = true` через aging не вызывался `dropCarriedItem()`.

**Исправление:**
```js
this.dropCarriedItem(); // до установки dead = true
this.dead = true;
this.state = 'dead';
```

**Архитектурный урок:**
Любой путь смерти (урон, старость, будущие эффекты) должен вызывать `dropCarriedItem()`. Хороший кандидат для метода `die()`, централизующего все инварианты смерти.

---

### Баг 12 — Разведчик подбирал ресурс при приземлении (логический)

**Файл:** `Minion.js` → `onSettle()`

**Симптом:** После броска разведчика рукой при приземлении рядом с ресурсом он мог автоматически войти в состояние `busy` и начать его собирать — вопреки роли разведчика.

**Причина:** В `onSettle()` не было проверки `goblinClass === 'scout'` — логика auto-gather срабатывала для всех гоблинов кроме воина и скелета.

**Исправление:** Добавлена явная ветка после проверки воина:
```js
if (this.goblinClass === 'scout') {
    this.pickNewTarget();
    this.state = 'free';
    this.stateTime = 0;
    return;
}
```

**Архитектурный урок:**
При добавлении нового класса юнита нужно явно проверить все точки входа в задачи: `onSettle()`, `input.js` флаг+ресурс, `input.js` флаг+место. Класс с особым поведением (не-сборщик) должен быть явно исключён во всех этих местах.

### Баг 13 — Мёртвые миньоны оставались в selectedMinions (логический)

**Файл:** `main.js` → `update()`, управление `hand.selectedMinions`

**Симптом:** Если разведчик (или любой гоблин) умирал пока был выделен, его индекс оставался в `hand.selectedMinions`. Клик по иконке в selection bar выполнял `m.state = 'free'` на мёртвом миньоне.

**Причина:** Очистка `selectedMinions` происходила только при явном действии игрока (снятие флага, клик по иконке), но не при смерти миньона.

**Исправление:** После цикла обновления миньонов — фильтрация мёртвых/нежити из `selectedMinions`. При пустом списке — автоматическое снятие флага:
```js
if (hand.grabbedFlag && hand.selectedMinions.length > 0) {
    hand.selectedMinions = hand.selectedMinions.filter(idx => {
        const m = minions[idx];
        return m && !m.dead && !m.isUndead;
    });
    if (hand.selectedMinions.length === 0) {
        flag.state = 'docked';
        hand.grabbedFlag = false;
        // ...
    }
}
```

**Архитектурный урок:**
`selectedMinions` — производное состояние от `minions[]`. Его нужно синхронизировать при каждом изменении `minions[]`: смерть, splice (уже есть), смена класса. Альтернатива — хранить ссылки на объекты, а не индексы, и фильтровать по `m.dead || m.isUndead`.
